# CHANGELOG 同步契约

CHANGELOG 面向评估升级风险的人类读者。优先服从项目既有发布约定；首次创建时采用 Keep a Changelog 与项目实际版本方案。

## 结构

- `[Unreleased]` 位于所有已发布版本之前。
- 已发布版本按新到旧排列，标题包含项目采用的版本号与 ISO 日期。
- 使用 `Added | Changed | Deprecated | Removed | Fixed | Security` 中实际有内容的分类；空分类不保留。
- 项目已有稳定扩展分类时继续使用，不为单次条目新造分类。
- 版本链接或 compare 链接必须指向真实仓库、最新 tag 和正确范围。

已发布条目是历史记录，不根据当前实现重写其措辞。只修复客观错误的版本号、日期或链接，并在报告中说明。

## 从 Git 区间提炼

1. 使用 state 的 `last_range.from_sha/to_sha` 读取完整 diff，而不是把 `git log` 逐条倾倒。
2. 按用户可感知的能力、行为、兼容性、修复和安全影响聚合主题；测试、格式化和纯内部重构默认不写。
3. 每条回答“用户之前会遇到什么，现在发生了什么”；实现细节只在解释影响所必需时出现。
4. 删除 `[Unreleased]` 中已被撤销、被替代、重复或已进入正式版本的内容。
5. 依赖升级只有在安全、兼容性或用户环境受影响时记录；已知 CVE 进入 `Security`。

Conventional Commit 只是导航信号，最终分类必须由实际 diff 和公共行为决定。

## 破坏性变更与发版

- 破坏性变更放在目标版本最显眼位置，明确影响范围、旧行为、新行为和可执行迁移步骤。
- 弃用必须给出替代入口和移除条件；没有迁移路径时不得只写“已弃用”。
- 只有真实版本变更和 tag 存在时，才把 `[Unreleased]` 内容迁移到正式版本；tag 后的新 commit 留在新的 `[Unreleased]`。
- Release notes 可以从 CHANGELOG 派生，但不能反向覆盖人工维护的历史。

## 条目质量

- 使用项目语言、主动语态和具体模块/能力名。
- 命令、标志、路径、配置键和版本号使用代码格式。
- 合并同一主题的多个 commit；删除 merge、WIP、格式化和“多项改进”等噪音。
- 文档自身变化只有在影响用户采用、迁移或接口理解时才记录。

完成标准：读者可以判断是否升级、会受什么影响以及如何迁移；区间内全部用户可感知变化已覆盖，内部噪音未进入历史。

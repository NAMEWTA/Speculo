# 文档生命周期契约

Diff 只负责发现变化；文档结论必须来自当前源码、配置、测试、CI、发布元数据、workflow 声明和已确认归档证据。

## 映射与整份审计

1. 把输入区间按对外能力、CLI/API、配置、架构、依赖、测试、发布、workflow 和文档自身分组。
2. 映射到确认范围内的文档；没有 owner 的候选先进入 `propose-only`，不得临时扩大范围。
3. 一旦文档被命中，读取整份文件和直接事实源，逐段判断是否仍准确、仍有读者价值、是否与别处重复。
4. 为每个资产选择唯一动作：
   - `add`：存在已实现事实但没有承载位置。
   - `update`：当前说明错误、不完整或接口已变化。
   - `delete`：内容已失效且没有历史保留价值。
   - `merge`：多个位置表达同一事实，收敛为一个权威来源。
   - `keep`：复核后仍准确。
   - `propose-only`：范围、语义、所有权或破坏性动作未确认。
5. 报告记录动作、证据、目标和验证，不把每个 commit 机械转成文档条目。

同一文档内删除过期段落、旧命令、坏链接和重复说明属于常规同步；删除整个文件或目录必须逐次确认。禁止只在末尾追加新事实而保留被推翻的旧内容。

## 文档类型

- 教程回答“如何学习”，操作指南回答“如何完成任务”，参考回答“接口是什么”，解释回答“为什么如此设计”。单篇文档保持主导类型，跨类型内容改成链接。
- API、配置、架构、迁移和发布说明必须与可执行入口或 schema 逐项核对。
- 示例必须可复制执行并展示必要的预期结果；不能运行时明确记录原因和静态证据。
- 多语言镜像保持章节、代码实体、命令、URL 和关键约束对等；只翻译说明文字。
- 历史版本、ADR 和已发布记录按项目规则归档，不把当前实现反写成虚假历史。

## 验证

至少执行：

- 项目声明的 build/test/lint 或等价质量闸。
- Markdown 相对链接、标题层级和代码块完整性检查。
- 文档中的命令、版本、路径、环境变量、配置键、CLI/API 表与事实源交叉检查。
- 全局搜索旧名称、删除路径、过期版本和被替换措辞，确认没有残留或重复事实源。

完成标准：所有命中资产已穷尽审计且动作唯一；新增、修改和删除均有当前证据；验证通过或运行已无损阻塞。

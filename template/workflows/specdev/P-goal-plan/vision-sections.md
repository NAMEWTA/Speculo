# 远景章节 §1-3 编写规程

编写 goal-plan 的前三个章节——定义"做什么"和"怎么算做完"。逐节草拟，每节经用户确认后再进入下一节。不一口气抛出全部三节。

## §1 — Goal

### 五要素公式

从 spec.md 的 Problem Statement 和 Solution 段落提取，合成一段密集声明，包含五个要素：

1. **构建什么**（what）——从 spec.md Solution 的第一句提炼
2. **不可协商的约束**（constraints）——从 ADR.md 架构决策中提取，至少一条
3. **具体交付物**（deliverable）——ticket 数量 + issue 范围（如 "#18-#38，共 21 个 issues"）
4. **质量标杆**（quality bar）——如果激活参考权威模式，声明与参考权威的对应关系（如"与 openhanako 手感一致"）；否则从 spec.md 的 Test Decisions 提取
5. **反模式**（anti-pattern）——声明禁止的伪完成（如"能力存在但手感不像 X"）

### 编写规则

- 一段写到底，不分点、不用列表
- 每个要素是可检查的声明，不是模糊愿望
- 反模式必须具体到可识别——禁止"代码写完但交互细节不匹配"

### 草拟与确认

先输出 §1 草案，等待用户确认。确认后进入 §2。

## §2 — Authoritative Inputs

### 优先级表

构建一张三列表格：优先级 | 文件路径 | 角色说明。行按优先级降序排列。

按检测到的模式逐行添加：

1. **合同行**（如激活合同模式）——优先级 1，路径指向合同文档，角色："冻结合同——含编号验收条目，先于所有其他参考"
2. **参考权威行**（如激活参考权威模式）——优先级 2，路径指向参考快照，角色："UX 真理——冻结快照，不随上游漂移。手感比对基准"
3. **ADR 行**——从 ADR.md 和永久 ADR 目录（`<Path>{roots.state}/specdev/adr/</Path>`）提取，每份一个行，优先级按 ADR 编号
4. **领域文档行**——从 CONTEXT.md 提取，每份一个行，角色："领域词汇表——术语定义和弃用同义词的权威来源"
5. **Spec/Ticket 行**——spec.md 和 tickets-map.md，优先级最低

### 冲突裁决顺序

从 input-validation.md 的推导结果复制，写为编号列表。

### 锁定产品裁定

从 ADR.md 的架构决策和 spec.md 的 Implementation Decisions 中提取不可推翻的裁定。每条裁定写为一句声明，标注来源（ADR 编号或 Issue 编号）。

- 裁定必须是已做出且不可推翻的——只提取 ADR 中标记为"决定"状态的条目
- 如果激活偏差模式，在此声明偏差编号和每条偏差的理由

### 草拟与确认

先输出 §2 完整草案（优先级表 + 冲突顺序 + 锁定裁定），等待用户确认。确认后进入 §3。

## §3 — Definition of Done

### 六道门禁骨架

逐道门禁填空，每道门禁写为一条可客观验证的全部或无声明：

1. **Issue 全绿**——填写 ticket 数量和仓库引用。格式：「所有 <N> 个 issues 全部关闭，每个 issue 的 checklist 全部勾选为 [x]」
2. **合同/ADR 状态收敛**——如果激活合同模式：填写合同文档名和待验收条目数。格式：「`<合同>` 中所有 <P0|P1|P2> 条目状态非 `todo`——`done` 或 `deviate`（有记录理由）」；如果无合同：填写 ADR 编号。格式：「ADR-<NNNN> 状态保持 `accepted`，偏差表条目已解决或明确记录为 `deviate`」
3. **验证门禁绿**——从 spec.md 的 Test Decisions 提取验证命令。格式：「`<verify_cmd>` 全绿」
4. **架构不回退**——填写 lint 和类型检查。格式：「无架构退化——<lint rules>」
5. **已锁裁定不松动**——引用 §2 的锁定产品裁定。格式：「§2 的 <N> 条锁定产品裁定全部满足，无静默推翻」
6. **可追溯**——填写 commit 到 issue 的链接要求。格式：「每个关闭的 issue 包含实现 commit 引用和验证截图/日志」

### 填充规则

- ticket 数量从 tickets-map.md 统计
- 合同条目从合同文档的表格行计数
- verify 命令从 spec.md Test Decisions 提取，如无则使用 `pnpm verify`
- 锁定裁定数量从 §2 提取

### 草拟与确认

输出 §3 完整草案，等待用户确认。

**完成标准**：§1 目标声明五要素齐全且经用户确认；§2 优先级表、冲突裁决顺序、锁定裁定完整且经用户确认；§3 六道门禁全部填充为可客观验证的声明且经用户确认。

# 输入验证与模式检测

在开始编写 goal-plan 任何章节之前，验证上游输入完整性并检测适用的编排模式。模式检测在第一步完成，后续所有章节根据检测结果适配内容。

## 1. 必需输入检查

逐项检查以下输入是否存在，缺失时按规则处理：

| 输入 | 路径 | 缺失处理 |
|------|------|----------|
| spec.md | `<Path>{roots.state}/specdev/changes/{change}/spec.md</Path>` | **中止**——引导用户先运行 `S-spec`，无 spec 无法定义目标 |
| tickets-map.md | `<Path>{roots.state}/specdev/changes/{change}/tickets-map.md</Path>` | **中止**——引导用户先运行 `T-tickets`，无 tickets 无法构建 DAG |
| ADR.md | `<Path>{roots.state}/specdev/changes/{change}/ADR.md</Path>` | 继续但记录——约束推导将缺少架构锚点 |
| CONTEXT.md | `<Path>{roots.state}/specdev/changes/{change}/CONTEXT.md</Path>` | 继续但记录——术语将草拟而非引用 |
| LOG.md | `<Path>{roots.state}/specdev/changes/{change}/LOG.md</Path>` | 继续但记录——设计轨迹不可追溯 |

如果 spec.md 和 tickets-map.md 均存在，继续下一步。任何必需项缺失且用户拒绝先运行上游 work，则中止。

## 2. 模式检测

对以下五个问题逐一回答，每个"是"激活对应的编排模式：

### Q1：是否存在冻结合同/验收文档？

在 spec.md 和 ADR.md 中搜索合同/验收文档引用。合同是包含编号验收条目（P0-01、P1-01 等）及状态值（`todo`/`done`/`deviate`/`n/a`）的外部文档。

- **是** → 激活「冻结合同模式」：§2 权威输入优先级表增加合同行；§3 DoD 增加合同状态收敛条目；§5 执行协议包含回写合同步骤；§6 里程碑验收包含合同终审；§7 硬约束包含合同冻结
- **否** → 使用简化模式：DoD 不含合同条目；执行协议不含回写合同；约束以 ADR 为权威

### Q2：是否存在参考权威（UX 快照、既有代码库）？

在 spec.md 的"解决方案"或"参考"段落中搜索对快照/代码库/产品的引用。

- **是** → 激活「参考权威模式」：§2 权威输入优先级表增加参考权威行；§5 执行协议步骤 1 包含对照路径；§6 人工 side-by-side 清单增加参考权威对比列
- **否** → 无参考权威：跳过参考对比相关内容

### Q3：是否存在有意偏差（DEV-01...）？

在合同文档或 ADR 中搜索偏差表（DEV-01 及更高编号）。偏差是预先批准的对参考权威的偏离。

- **是** → 激活「偏差模式」：§2 权威输入增加偏差表引用；§3 DoD 增加偏差收敛条目；§5 执行协议步骤 6 包含偏差追踪
- **否** → 无偏差追踪需求

### Q4：ticket 数量在哪个范围？

统计 tickets-map.md 中的 ticket 条目数：

- **< 10 个** → 「轻量模式」：警告用户 goal-plan 为 10+ tickets 设计，提供两个选项：(a) 继续用简化模板（一步编写全部章节），(b) 跳过 goal-plan 直接进入 I-implement
- **10-20 个** → 「标准模式」：启用并发控制（max 3），DAG 中等复杂度
- **> 20 个** → 「复杂模式」：完整并发控制、严格门禁次序（P0→P1→P2）、§9 速查表强制追加

### Q5：采用何种执行模型？

检查用户是否有意使用 Lead+Subagent 模型。默认如果 ticket 数 ≥ 10 或有合同文档，推荐 Lead+Subagent。

- **Lead+Subagent 模型** → §5 执行协议使用完整八步流程（读取→派单→实现→双轴审查→门禁→回写→关闭→纪律）；§7 硬约束包含 Lead 纪律规则；§8 进度回报使用 TICKET_DONE 格式
- **简化模型** → §5 使用精简协议（读取→实现→审查→门禁→关闭）；不使用 TICKET_DONE 格式

## 3. 冲突裁决顺序

从检测到的模式推导冲突裁决顺序，默认顺序（从高到低）：

1. 领域不变量（来自 ADR.md 架构决策）
2. 冻结合同（如激活合同模式）
3. 参考权威手感（如激活参考权威模式）
4. spec.md 中的用户故事

如果未激活合同模式，跳过第 2 项。如果未激活参考权威模式，跳过第 3 项。

## 4. 输出模式检测摘要

将检测结果汇总为结构化摘要，呈现给用户确认：

```
## 模式检测摘要

| 检测项 | 结果 | 影响 |
|--------|------|------|
| 冻结合同 | 是/否 | ... |
| 参考权威 | 是/否 | ... |
| 有意偏差 | 是（DEV-01 至 DEV-0X）/ 否 | ... |
| Ticket 数量 | N（轻量/标准/复杂） | ... |
| 执行模型 | Lead+Subagent / 简化 | ... |
| ADR 可用 | 是/否 | ... |
| CONTEXT 可用 | 是/否 | ... |

冲突裁决顺序：[推导结果]

是否确认以上模式检测？如有误请纠正后继续。
```

用户确认或纠正后，进入远景章节编写。

**完成标准**：五个模式检测问题均已回答；冲突裁决顺序已推导；摘要已呈现并获用户确认或纠正。

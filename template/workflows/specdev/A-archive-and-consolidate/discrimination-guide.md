# 知识鉴别指南

将已完成变更中通过毕业标准的知识，与三个永久知识库（`adr/`、`context/`、`research/`）的现有内容逐项比对，判定处置动作。**核心原则：不盲目追加——每次写入必须经过比对评估。**

## 四步鉴别法

### 第一步：读取现有知识库

对三个知识库建立可检索索引：

- **adr/**：逐文件读取 `<Path>{roots.state}/specdev/adr/</Path>` 下所有 `NNNN-slug.md`，提取标题、决策主题、状态（Accepted/Superseded/Deprecated）、Superseded 链
- **context/**：逐文件读取 `<Path>{roots.state}/specdev/context/</Path>` 下所有术语定义，提取术语名、定义文本、`_Avoid_` 列表
- **research/**：读取 `<Path>{roots.state}/specdev/research/index.md</Path>` 索引表及所有研究文件，提取主题、结论摘要、来源变更

### 第二步：逐项比对

将每个知识候选项与对应目标库的现有内容进行比对：

#### adr/ 比对

对每个 ADR 候选项，在现有 `adr/` 中按**决策主题**（非标题字符串）匹配：

| 匹配结果 | 处置动作 | 说明 |
|---------|---------|------|
| 主题未找到 | **create** | 创建新 ADR 文件，分配下一个可用序号（四位零填充） |
| 主题相同，候选项更全面或更新了结论 | **supersede** | 旧 ADR 添加 `Superseded by ADR-NNNN` 横幅；创建新 ADR |
| 主题相同，候选项补充了新后果或上下文 | **update** | 将新信息合并到现有 ADR 正文，补充 Consequences 或 Context |
| 主题相同，内容实质一致 | **skip** | 标注"已有"，不重复写入 |
| 主题相同，结论直接矛盾 | **needs-confirmation** | 展示双方版本及建议，等待用户裁决 |
| 现有 ADR 已被新变更推翻且 >30 天无引用 | **retire** | 标记废弃，移入清理候选（在清理计划中处理） |

**Supersede 链处理**：若发现链式取代（A 被 B 取代，B 被 C 取代），C 为当前版本，A 和 B 为历史。只对链末端操作。

#### context/ 比对

对每个术语候选项，在现有 `context/` 中按**术语名**和 **`_Avoid_` 别名**精确匹配：

| 匹配结果 | 处置动作 | 说明 |
|---------|---------|------|
| 术语名和别名均未找到 | **create** | 添加新术语条目（`**术语**：定义` + `_Avoid_` 列表） |
| 术语已存在，定义相同 | **skip** | 标注"已有"，不重复 |
| 术语已存在，定义更精确或范围扩展 | **update** | 合并更新定义；`_Avoid_` 列表取并集 |
| 术语已存在，定义实质不同 | **needs-confirmation** | 展示双方版本及建议，等待用户裁决 |
| 术语被重命名（旧名 → 新名） | **rename** | 创建新术语条目，将旧名加入新条目的 `_Avoid_`；旧条目保留并标注指向新条目 |
| 现有术语在所有活跃变更和代码中均无引用 | **retire** | 弱信号——标记为待用户审核，不自动删除 |

**术语合并规则**：
- 同一概念的不同表述 → 指定更精确的定义为权威版本
- 定义相同但 `_Avoid_` 列表有补充 → 合并 `_Avoid_`（取并集）
- 分组维护：若术语属于已有分组，归入对应分组

#### research/ 比对

对每个研究候选项，在 `<Path>{roots.state}/specdev/research/index.md</Path>` 中按**主题**匹配：

| 匹配结果 | 处置动作 | 说明 |
|---------|---------|------|
| 主题未找到 | **create** | 复制研究文件到 `_state/research/`，更新 `index.md` |
| 主题已存在，新发现补充 | **update** | 合并新发现到现有研究文件，更新日期和来源 |
| 主题已存在，新研究完全取代 | **supersede** | 标注旧研究为 superseded，新研究成为权威版本 |
| 主题已存在，发现一致 | **skip** | 标注"已有"，不重复 |
| 研究发现矛盾 | **needs-confirmation** | 展示双方结论，等待用户裁决 |
| 纯变更特定（仅对该变更有用） | **skip** | 保留在归档变更中，不提升 |

### 第三步：交叉验证

- 一个变更的知识可能关联多个现有条目——检查是否遗漏关联
- 新 ADR 可能影响多个现有 context 术语——检查术语是否需要同步更新
- 研究结论可能支持或削弱现有 ADR——检查是否需要标注关联

### 第四步：标注处置

每个知识项输出以下信息：

```
- 来源变更：<change-name>
- 来源文件：ADR.md / CONTEXT.md / LOG.md / research/<topic>.md
- 知识类型：架构决策 / 领域术语 / 研究产物
- 目标知识库：adr/ / context/ / research/
- 处置动作：create / update / merge / supersede / retire / skip / needs-confirmation
- 毕业标准：稳定机制 / 重复教训 / 接手者必知
- 内容摘要：1-2 句话
- 风险等级：低 / 中 / 高
```

## 处置动作汇总

| 动作 | 含义 | 写入行为 |
|------|------|---------|
| **create** | 全新知识，知识库中不存在 | 创建新文件或新条目 |
| **update** | 已有知识，候选项更精确/全面 | 合并写入现有文件 |
| **merge** | 实质相同但表述不同 | 保留现有为权威，补充候选项独特内容 |
| **supersede** | 候选项取代现有知识 | 旧条目加横幅，新条目创建 |
| **retire** | 现有知识已过时 | 标记废弃，移入清理候选 |
| **skip** | 完全相同或不符合毕业标准 | 不写入，记录原因 |
| **needs-confirmation** | 冲突或实质性矛盾 | 展示双方版本，等待用户裁决 |

# UI 原型

在单个路由上生成**几个截然不同的 UI 变体**，通过底部浮动栏切换。用户在浏览器中翻看变体，选一个（或从每个中偷一些元素），然后丢弃其余。

如果问题是关于逻辑/状态而非界面外观——选错了分支。用 [LOGIC.md](LOGIC.md)。

## 适用场景

- "这个页面应该长什么样？"
- "我想在提交之前看几个仪表盘方案。"
- "给设置页试一种不同的布局。"
- 任何用户本来会在脑子里花一天时间在三个模糊线框图之间犹豫不决的场景。

## 两种子形态 —— 强烈偏好子形态 A

UI 原型在**与应用的其余部分产生摩擦**时才最容易评判——真实的 header、真实的 sidebar、真实的数据、真实的信息密度。单独的一次性路由是真空：每个变体在隔离状态下看起来都不错。只要有合理的现有页面可以承载变体，就默认使用子形态 A。只有当原型确实没有邻近的宿主时才使用子形态 B。

### 子形态 A — 调整现有页面（首选）

路由已存在。变体在**同一路由**上渲染，通过 `?variant=` URL 查询参数控制。现有的数据获取、参数和认证全部保留——只替换渲染部分。这是默认选项；除非有明确的理由不这样做，否则选它。

如果原型针对的东西还没有页面，但*自然地应该存在于某个页面内部*（仪表盘的新区域、设置页的新卡片、现有流程中的新步骤）——这仍然是子形态 A。将变体挂载在宿主页面内部。

### 子形态 B — 新建页面（最后手段）

仅当被原型化的事物确实没有现成页面可以嵌入时使用——例如一个全新的顶层界面，或一个无法合理嵌入任何地方的流程。

按照项目已有的路由约定创建一个**一次性路由**——不要发明新的顶层结构。命名要让人一眼看出是原型（例如在路径或文件名中包含 `prototype` 字样）。同样使用 `?variant=` 模式。

在提交子形态 B 之前，做一个合理性检查：真的没有现成页面可以嵌入吗？空路由会隐藏有内容的页面能够暴露的设计问题。

两种子形态下，底部浮动栏完全相同。

## 流程

### 1. 陈述问题并确定变体数量 N

默认 **3 个变体**。超过 5 个就不再是截然不同，而是噪音——以此为上限。

将计划写在一行内，放在原型所在位置或文件顶部注释中：

> "设置页的三个变体，通过 `?variant=` 切换，在现有 `/settings` 路由上。"

无论用户是否在场反对，这都能成立。

### 2. 生成截然不同的变体

起草每个变体。每个变体必须满足：

- 页面的目的和它能访问的数据。
- 项目的组件库 / 样式系统（TailwindCSS、shadcn、MUI、纯 CSS，等等）。
- 清晰的导出组件名，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上不同**——不同的布局、不同的信息层次、不同的主要操作入口，而不仅仅是不同的颜色。三个微调过的卡片网格不是 UI 原型，是壁纸。如果两份草稿太相似，用明确的"不要用卡片网格"指引重做其中一个。

### 3. 将它们串接起来

在路由上创建一个单一的切换器组件：

```tsx
// 伪代码 —— 根据项目框架调整
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

对于子形态 A（现有页面）：将现有数据获取保持在切换器上方；每个变体只替换渲染的子树。

对于子形态 B（新建页面）：`/prototype/<名称>` 下的一次性路由挂载同一个切换器。

### 4. 构建浮动切换器

一个位于屏幕底部中央的固定定位小栏，包含三个元素：

- **左箭头**——切换到上一个变体（循环）。
- **变体标签**——显示当前变体标识，如果变体导出了名称，也显示名称。例如 `B — 侧边栏布局`。
- **右箭头**——切换到下一个（循环）。

行为：

- 点击箭头更新 URL 查询参数（使用框架的路由器——Next 上用 `router.replace`、React Router 上用 `navigate`，等等），使变体可分享且在刷新后保持。
- 键盘：`←` 和 `→` 方向键也可切换。当 `<input>`、`<textarea>` 或 `[contenteditable]` 元素聚焦时不要拦截方向键。
- 在视觉上与页面区分（如高对比度胶囊形、微妙阴影），使其明显不是被评估的设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等价检查进行门控，这样即使原型不小心合入也不会把切换器发布给用户。

将切换器放在一个共享组件中，供两种子形态复用。放置在项目中共享 UI 组件的通常位置。

### 5. 交付

给出 URL（以及 `?variant=` 的各个键值）。用户会在有空时翻看。最有趣的反馈通常是**"我想要 B 方案的头和 C 方案的侧边栏"**——那才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦某个变体胜出，按 [SKILL](SKILL.md) 中持久化约定的方式捕获答案：

1. **融入正式代码**：
   - **子形态 A** — 将胜出变体融入现有页面；从主分支移除落选变体和切换器。
   - **子形态 B** — 将胜出变体提升为正式路由；从主分支移除一次性路由和切换器。
2. **持久化评估记录**：在 `<Path>{roots.state}/<workflow>/changes/{change}/prototype/ui-<topic>.md</Path>` 创建答案文件，记录：
   - 所回答的 UI 问题
   - 哪个变体胜出及原因——完整的评估推理
   - 各变体的结构差异分析
   - 从落选变体中提取的有价值元素（如果适用）
   - throwaway 分支指针
3. **UI 规范沉淀**：将评估过程中产生的 UI 规范洞察（如布局原则、信息层次、交互模式选择理由）写入答案文件，供后续 spec 编写引用。
4. **清理原型代码**：将完整变体集（包括落选变体和切换器）提交到 throwaway 分支，不进入主分支。变体组件和切换器留在主分支会快速腐烂并误导后续读者。
5. **更新索引**：将新答案追加到 `prototype/index.md` 表格中。

## 反模式

- **变体仅颜色或文案不同。** 那是微调，不是原型。真正的变体在结构上存在分歧。
- **变体之间共享过多代码。** 共享一个 `<Header>` 没问题；共享一个 `<Layout>` 就失去了意义。每个变体应该能够自由地抛弃布局。
- **将变体接入真实的数据变更。** 只读原型完全没问题。如果变体需要变更数据，将其指向一个桩——问题是"这个应该长什么样"，不是"后端是否正常工作"。
- **将原型直接提升到生产环境。** 变体代码是在原型约束下编写的（无测试、最小错误处理）。融入时要正确重写。

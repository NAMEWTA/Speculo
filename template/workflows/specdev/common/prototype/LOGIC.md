# 逻辑原型

构建一个微小的交互式终端应用，让用户手动驱动状态模型。当问题涉及**业务逻辑、状态转换或数据形态**时使用——这类问题在纸面上看起来合理，但只有推进真实用例后才会暴露出不对劲的地方。

## 适用场景

- "我不确定这个状态机能否处理先 X 后 Y 的边界情况。"
- "这个数据模型真的能表示那种情况吗……"
- "我想在写之前先感受一下 API 应该长什么样。"
- 任何用户想要**按按钮、观察状态变化**的场景。

如果问题是"这个应该长什么样"——选错了分支。用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

在写代码之前，写下你正在为哪个状态模型和哪个问题做原型。一段话即可，放在原型的 README 或文件顶部的注释中。回答了错误问题的逻辑原型是纯粹浪费——让问题显式化，这样之后可以核查，无论用户是现在看着还是稍后 AFK 回来再看。

### 2. 选择语言

使用宿主项目所用的语言。如果项目没有明显的运行时（如文档仓库），则询问。

遵循项目已有的工具链约定——不要仅为原型引入新的包管理器或运行时。

### 3. 将逻辑隔离到一个可移植模块中

将实际逻辑——回答问题的部分——放在一个小巧、纯净的接口后面，使其之后可以被提取并放入正式代码库。围绕它的 TUI 是一次性的；逻辑模块不应该是一次性的。

正确的形态取决于问题：

- **纯 reducer**——`(state, action) => state`。适用于动作为离散事件且状态为单一值的场景。
- **状态机**——显式的状态和转换。适用于"当前哪些操作是合法的"本身就是问题的一部分。
- **一组纯函数**操作一个纯数据类型。适用于没有隐式当前状态、只有转换的场景。
- **类或模块**——具有清晰方法接口，当逻辑确实拥有持续性内部状态时使用。

选择最适合所问问题的形态，而*不是*最容易接入 TUI 的形态。保持纯净：无 I/O、无终端代码、无用于控制流的 `console.log`。TUI 导入它并调用它；反向不传递任何内容。

这就是让原型在自身生命周期之后仍有价值的关键：当问题得到回答后，验证通过的 reducer / 状态机 / 函数集可以被单独提升到正式模块中。

### 4. 构建最小的 TUI 来暴露状态

将其构建为**轻量 TUI**——每次 tick 清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价方式）并重新渲染整个帧。用户应始终看到一个稳定视图，而非不断增长的滚动回溯。

每帧包含两部分，顺序如下：

1. **当前状态**，pretty-print 且 diff 友好（每行一个字段，或格式化 JSON）。使用**粗体**标注字段名或节标题，**暗色**标注次要上下文（时间戳、ID、派生值）。原生 ANSI 转义码即可——`\x1b[1m` 粗体、`\x1b[2m` 暗色、`\x1b[0m` 重置。无需引入样式库，除非项目中已经存在。
2. **键盘快捷键**，列在底部：`[a] 添加用户  [d] 删除用户  [t] 推动时钟  [q] 退出`。粗体标键、暗色标描述，或反过来——怎么读起来清晰怎么来。

行为：

1. **初始化状态**——单个内存中的对象/结构体。启动时渲染第一帧。
2. **每次读取一次按键（或一行）**，分发到修改状态的处理器。
3. **每次操作后重新渲染**完整帧——不追加，而是替换。
4. **循环直到退出。**

整个帧应适配一屏。

### 5. 一条命令即可运行

向项目已有任务运行器添加一条脚本（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应运行 `pnpm run <原型名称>` 或等价命令——永远不需要记住路径。

如果宿主项目没有任务运行器，直接把命令写在原型 README 的顶部。

### 6. 交付

给用户运行命令。他们会自己驱动它；有趣的时刻是他们说"等等，那不应该可能"或"嗯，我以为 X 会不一样"——那些是_想法_中的 bug，这正是整个原型的目的。如果他们想添加新操作，就添加。原型会演化。

### 7. 捕获答案并持久化

原型回答问题后，按 [SKILL](SKILL.md) 中持久化约定的方式捕获答案：

1. **提升验证过的逻辑**：将验证通过的 reducer / 状态机 / 函数集提升到正式模块中（决策已被吸收）。
2. **持久化答案记录**：在 `<Path>{roots.state}/<workflow>/changes/{change}/prototype/logic-<topic>.md</Path>` 创建答案文件，记录：
   - 所回答的问题
   - 结论——什么可行、什么不可行
   - 被验证的逻辑模块的描述
   - throwaway 分支指针（TUI 外壳代码所在位置）
3. **更新索引**：将新答案追加到 `prototype/index.md` 表格中。

TUI 外壳代码仍提交到 throwaway 分支——它是一次性的交互壳，真正有价值的部分（逻辑模块）已经提升到正式代码中。

## 反模式

- **不要加测试。** 需要测试的原型不再是原型。
- **不要接入真实数据库。** 使用内存存储，除非问题本身就是关于持久化的。
- **不要泛化。** 不要"如果我们以后想支持 X 呢"。原型只回答一个问题。
- **不要把逻辑和 TUI 混在一起。** 如果 reducer / 状态机引用了 `console.log`、提示符或终端转义码，它就不可移植了。让 TUI 成为纯模块外面的薄壳。
- **不要把 TUI 外壳发布到生产环境。** 外壳是为在终端中手动驱动而优化的。背后的逻辑模块才是值得保留的部分。

# 反馈回路构建技术

反馈回路是诊断的核心——一个紧凑的通过/失败信号，锚定 bug 的症状。以下技术按大致优先级排列，从最理想到最后手段。

## 构建方式

按顺序尝试，直到获得一个可工作的回路：

### 1. 失败测试

在能触及 bug 的任意缝合点编写——单元测试、集成测试、端到端测试。优先选择现有的测试缝合点。观察项目中已有的测试了解如何注入测试。

### 2. Curl / HTTP 脚本

针对正在运行的开发服务器。固定请求参数，对响应状态码和关键字段进行断言。

### 3. CLI 调用

带固定输入调用 CLI，将 stdout 与已知良好快照进行 diff。适合命令行工具和数据管道。

### 4. 无头浏览器脚本

Playwright 或 Puppeteer 驱动 UI，对 DOM 状态、控制台输出、网络请求进行断言。适合前端 bug。

### 5. 回放捕获的追踪数据

将真实网络请求、负载、事件日志保存到磁盘，在隔离环境中通过代码路径回放。适合需要真实数据的场景。

### 6. 一次性测试夹具

启动系统的最小子集——一个服务、模拟依赖——用单个函数调用驱动 bug 代码路径。适合微服务或多组件交互的 bug。

### 7. 属性 / 模糊测试循环

如果 bug 表现为"有时输出错误"，运行 1000 次随机输入寻找故障模式。适合数据相关或边界条件 bug。

### 8. 二分查找夹具

如果 bug 出现在两个已知状态之间（commit、数据集、版本），自动化"在状态 X 启动、检查、重复"以便 `git bisect run`。

### 9. 差分循环

通过旧版本 vs 新版本（或两种配置）运行相同输入，对输出进行 diff。适合回归 bug。

### 10. HITL bash 脚本

最后手段。如果必须由人工点击，使用 `<Path>{roots.workflows}/specdev/common/scripts/hitl-loop.template.sh</Path>` 驱动人工操作，使循环仍然结构化。捕获的输出反馈给 agent。

复制模板，编辑步骤，运行脚本。脚本中的 `step` 函数显示指令并等待按 Enter，`capture` 函数显示问题并读取响应。结束时以 `KEY=VALUE` 格式打印捕获的值供 agent 解析。

## 收紧回路

将回路视为产品。一旦有了一个回路，收紧它：

- **更快**——缓存设置、跳过无关初始化、缩小测试范围
- **更清晰**——针对具体症状断言，而非"没崩溃"
- **更确定**——固定时间、种子随机数、隔离文件系统、冻结网络

一个 30 秒的抖动回路比没有回路好不了多少；一个 2 秒的确定性回路是调试的超能力。

## 非确定性 bug

目标不是干净的复现，而是更高的复现率。循环触发 100 次，并行化，增加压力，缩小时间窗口，注入 sleep。50% 抖动的 bug 是可调试的；1% 则不行——持续提高复现率直到可调试。

## 当确实无法构建回路时

停下来，明确说明。列出已尝试的方法。向用户请求：

1. 访问能复现的任何环境
2. 一个捕获的产物——HAR 文件、日志转储、核心转储、带时间戳的屏幕录制
3. 添加临时生产环境插桩的许可

在没有回路的情况下进入假设阶段是本规程要防止的核心失败模式。没有变红能力的命令，就没有后续阶段。

## 最小化协议

复现后，将场景缩小到仍能变红的最小场景。逐个削减以下元素，每次削减后重新运行回路：

1. 输入数据——减少字段、缩小数据集、简化参数
2. 调用者——移除中间层、直接调用核心逻辑
3. 配置——使用默认值、移除环境变量
4. 步骤——跳过前置操作、合并中间步骤

每个剩余元素必须有负载作用——移除其中任何一个都会使回路变绿。最小复现场景缩小了假设空间，并成为最终的回归测试基础。

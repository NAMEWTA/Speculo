# 插桩验证规则

每个探测必须映射到诊断计划中的一个具体预测。每次只改变一个变量。

## 探测映射

在添加任何插桩之前，明确陈述：

- 此探测验证诊断计划中的哪个假设、哪个预测
- 期望看到什么结果（如果假设正确）
- 期望看到什么结果（如果假设错误）

每次探测后记录实际结果，与预测对照。如果预测不匹配，该假设被排除，移至下一个。

## 工具偏好

按优先级选择插桩方式：

### 1. 调试器 / REPL 检查

如果环境支持，优先使用。一个断点胜过十行日志。在关键路径上设置条件断点，检查变量状态和调用栈。

### 2. 针对性日志

在能区分假设的边界处添加日志——模块接口、数据转换点、分支条件。日志内容应包含：

- 所在假设的简短标识
- 关键变量的值
- 与预测相关的状态信息

### 3. 每条日志必须有明确的验证目标

全量日志淹没信号。每条日志单独对应一个假设预测，通过唯一前缀追溯到诊断计划中的具体条目。

## 调试日志标记

所有调试日志使用唯一前缀标记，格式为 `[DEBUG-xxxx]`，其中 `xxxx` 为 4 位随机字母数字（如 `[DEBUG-a4f2]`）。在日志消息中包含：

```javascript
console.log(`[DEBUG-a4f2] 缓存键: ${cacheKey}, 用户ID: ${userId}`)
```

最终的清理只需一次 `grep`——未标记的日志保留，已标记的日志删除。

## 性能分支

对于性能回归，日志通常是错误工具。替代方案：

1. 建立基线测量——计时夹具、`performance.now()`、分析器、查询计划
2. 二分查找——定位引入回归的变更点
3. 先测量，后修复——在没有测量基线的情况下，任何优化都是猜测

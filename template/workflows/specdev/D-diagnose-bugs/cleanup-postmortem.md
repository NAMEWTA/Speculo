# 清理与事后分析

诊断完成、修复已由 I-implement 执行后，必须完成以下清理和复盘。

## 清理检查清单

逐项确认：

- [ ] 原始复现不再复现——重新运行阶段 2 的反馈回路，确认变绿
- [ ] 回归测试通过——I-implement 已为修复编写并通过回归测试
- [ ] 所有 `[DEBUG-xxxx]` 插桩已移除——`grep -r "\[DEBUG-"` 项目根目录，确认无残留
- [ ] 一次性原型已删除——移除所有为诊断创建的临时脚本、测试夹具、mock 数据
- [ ] 被证实的假设已记录到 `<Path>{roots.state}/specdev/changes/{change}/LOG.md</Path>`，以便下一个调试者学习

## 记录根因

在 `<Path>{roots.state}/specdev/changes/{change}/LOG.md</Path>` 中追加条目：

```markdown
## [YYYY-MM-DD] 诊断：<bug 简述>

- **根因**：<被证实的假设，包含技术细节>
- **验证方式**：<哪个探测确认了根因>
- **修复提交**：<I-implement 的提交 hash 或描述>
- **预防建议**：<什么本可以预防此 bug>
```

## 事后分析

问：什么本可以预防这个 bug？从以下维度审视：

- **测试缝合点**——是否缺少合适的测试接缝？如果有好的缝合点，此 bug 是否会被更早发现？
- **接口设计**——接口是否暴露了容易误用的契约？深度是否足够防止调用者犯错？
- **数据边界**——是否缺少输入校验、类型约束或边界条件处理？
- **耦合**——是否因模块间的隐藏耦合导致变更的连锁反应？

如果答案涉及架构变更（没有好的测试缝合点、纠缠的调用者、隐藏的耦合），将具体情况记录到 LOG.md 的预防建议中。在修复之后提出建议——此时比诊断开始时拥有更多信息。

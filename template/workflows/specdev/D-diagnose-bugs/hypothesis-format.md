# 假设格式与诊断计划

在插桩之前生成多个排名假设。单一假设会锚定在第一个看似合理的想法上。每个假设必须是可证伪的——陈述其预测。

## 假设格式

每个假设使用以下模板：

> 如果 `<根因>` 是原因，那么 `<改变 X>` 会使 bug 消失 / `<改变 Z>` 会使它更糟。

无法陈述预测的假设只是感觉——精炼或丢弃它。

示例：

> 如果缓存键在用户 ID 包含特殊字符时生成错误，那么将用户 ID 替换为纯字母数字字符串会使 bug 消失，而保持原 ID 并在缓存查询前打印键值会显示格式错误的键。

## 排名规则

生成 3-5 个假设，按以下优先级排名：

1. **最近变更**——最近修改的代码区域优先。新代码是 bug 最常见的来源。
2. **数据边界**——涉及空值、特殊字符、极限值、类型转换的假设优先于一般逻辑错误。
3. **时序与并发**——涉及竞态条件、异步顺序、超时的假设在单线程同步假设之后。
4. **外部依赖**——涉及第三方库、API 响应变更、环境差异的假设排在最后。

## 诊断计划呈现

将排名假设列表作为诊断计划呈现给用户。使用以下格式：

```markdown
## 诊断计划

基于当前症状和代码库理解，以下是 3-5 个按优先级排名的可证伪假设：

1. **[假设名称]** —— 如果 `<根因>` 是原因，那么 `<预测>`。
2. **[假设名称]** —— 如果 `<根因>` 是原因，那么 `<预测>`。
...

请确认计划或调整优先级。你是否已经排除了其中任何一个？
```

用户通常拥有能立即重排名的领域知识——低成本检查点，大幅节省时间。向用户展示后等待确认。

## AFK 默认行为

如果用户 AFK（无响应），按排名继续——从排名最高的假设开始插桩验证。在每个假设被排除后，更新排名并继续下一个，无需等待确认。

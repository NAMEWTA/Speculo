三层架构与数据流向

1. 宏观回顾

在进入细节之前，先建立一个心智模型。当一个用户说「帮我分类这些 issue」时，Speculo 内部发生了：

用户意图"帮我分类 issue"
→
ask-matt 路由命中 triage 路由
→
路由适配加载 vendor triage SKILL.md
将路径重定向到 state namespace
→
技能执行分类角色状态机运行
结果写入 changes/triage-xxx/
→
持久化status.json 更新
change 归档至 archive/

这就是三层的协作方式。每一层有自己的职责、自己的文件、自己的边界。下面逐层拆解。

2. 第 1 层：vendor/ — 原始技能资产

定位

vendor 层是 Matt Pocock 的原始技能代码。它在 Speculo 中的角色类似于第三方库——Speculo 使用它、编排它，但不修改它。

物理结构

vendor/matt-pocock/
├── README.md                    ← 技能清单（用户调用 vs 模型调用）
├── engineering/                 ← 编码技能
│   ├── ask-matt/SKILL.md        ← 路由器：根据意图推荐技能
│   ├── triage/SKILL.md          ← 分类 issue
│   ├── implement/SKILL.md       ← 构建实现
│   ├── to-spec/SKILL.md         ← 生成规范
│   ├── to-tickets/SKILL.md      ← 拆分工单
│   ├── code-review/SKILL.md     ← 代码审查
│   ├── diagnosing-bugs/SKILL.md ← 诊断 bug
│   ├── tdd/SKILL.md             ← 测试驱动开发
│   ├── prototype/SKILL.md       ← 一次性原型
│   ├── research/SKILL.md        ← 背景研究
│   ├── domain-modeling/SKILL.md ← 领域建模
│   ├── codebase-design/SKILL.md ← 代码仓设计
│   ├── wayfinder/SKILL.md       ← 大工作规划
│   ├── grill-with-docs/SKILL.md ← 文档化访谈
│   ├── improve-codebase-architecture/SKILL.md ← 架构改进
│   ├── resolving-merge-conflicts/SKILL.md     ← 合并冲突解决
│   └── setup-matt-pocock-skills/SKILL.md      ← 一次性配置
└── productivity/                ← 通用生产力技能
    ├── teach/SKILL.md           ← 跨会话教学
    ├── handoff/SKILL.md         ← 会话交接
    ├── grill-me/SKILL.md        ← 无状态访谈
    ├── grilling/SKILL.md        ← 访谈循环原语
    └── writing-great-skills/SKILL.md ← 技能编写指南

关键特征

  1. 每个技能一个目录，包含 SKILL.md 加可选的模板/格式/脚本
  2. 技能分为两种调用模式：
    • 用户调用：用户主动发起（如 triage、implement、teach）—— 在 WORKFLOW.md 的 routes 中映射
    • 模型调用：模型在内部自动使用（如 tdd、prototype、research）—— 不直接路由，由其他技能内部引用
  3. 完全自描述：技能不知道 Speculo 的存在。它们写「保存到当前目录」或「写入 .scratch/」，这些路径在编排层被适配重定向
  4. 不可修改：Speculo 使用者不应修改 vendor 中的技能文件。自定义应通过 workflow 层的路由适配实现

技能元数据格式

每个 SKILL.md 的 YAML frontmatter 定义了技能的接口契约：

---
name: teach
description: 在此工作区内教授用户一项新技能或概念。
disable-model-invocation: true    ← true = 仅用户调用
argument-hint: "你想学习什么？"
---

Workflow 层在路由时读取 name 和 description，disable-model-invocation 决定该技能是否可作为路由目标。

3. 第 2 层：workflows/ — 编排层

定位

编排层是 Speculo 的核心增值。它不定义新技能，而是用声明式语言描述：哪些用户意图触发哪些 vendor 技能、技能输出存到哪里、状态如何转移。如果说 vendor 是乐高积木，workflow 就是搭建图纸。

物理结构

workflows/matt-pocock/
├── WORKFLOW.md                  ← 主定义文件（编排核心）
└── routes/                      ← 路由定义文件
    ├── idea-to-delivery.md
    ├── wayfinder.md
    ├── triage.md
    ├── diagnose.md
    ├── architecture.md
    ├── review.md
    ├── merge-conflicts.md
    ├── research-prototype.md
    ├── productivity.md
    ├── experimental.md
    └── setup.md                 ← 特殊的懒配置路由

WORKFLOW.md 的 XML 声明式语法

主定义文件用 XML 块声明了 8 个核心概念。每个块是独立的领域描述：

XML 块角色核心语义
<runtime-context> 根别名映射 定义 workflow / state / vendor 三个根的物理路径别名，供后续所有块用别名引用而非硬编码路径
<persistence> 存储声明 声明 state 根下的存储骨架：哪些是固定骨架（index/changes/archive），哪些是懒创建（knowledge/integrations/backlog）
<routes> 路由表 10 条路由的优先级排序，每条绑定一个 <when> 触发条件和目标 route 文件
<sequence> 进入协议 三步启动序列：select-change → route → lazy-config，确保上下文就绪后才激活技能
<dependencies> 前置依赖 声明硬依赖（hard）—— 缺少则工作流不可用
<state-schema> 状态扩展字段 status.json 中除骨架字段外扩展的自定义字段及其类型
<transitions> 状态机 状态转移规则：route→phase、phase→route、active→archived
正文段落 持久化与副作用规则 路径重定向规则、外部副作用确认规则、临时文件清理规则

每个路由文件（routes/*.md）内部也使用类似的 XML 声明语法，定义该路由下的子路由、所需技能、产物路径和触发条件。

4. 第 3 层：.speculo/ — 运行时状态

定位

状态层是工作流的持久化存储。所有执行过程中的决策、产物、配置都沉淀在这里。它是 gitignored 的——不属于源码，属于运行实例。

物理结构

.speculo/
├── workspace.json              ← 项目根注册表（全局唯一）
├── person/                     ← 个人工作流状态
│   ├── status.json
│   ├── changes/
│   ├── archive/
│   └── .config/
├── matt-pocock/                ← Matt Pocock 工作流状态
│   ├── status.json             ← 当前状态：route、change、history
│   ├── changes/                ← 活跃 change 的产物
│   │   └── teach-20260711-190000/
│   │       └── teaching/       ← 当前 teach 技能的教学产物
│   ├── archive/                ← 已完成的 change
│   ├── knowledge/              ← 领域知识
│   │   └── domain.md
│   └── integrations/           ← 集成配置
│       ├── issue-tracker.md
│       └── triage-labels.md
└── commands/                   ← 命令定义

status.json 的骨架字段

{
  "schema_version": 1,         ← 格式版本（升级时迁移）
  "workflow": "matt-pocock",   ← 所属工作流
  "current_route": "productivity",  ← 当前路由
  "route_history": [...],      ← 路由跳转历史
  "active": [...],             ← 所有活跃 change 的 ID 列表
  "current_change": "teach-20260711-190000"  ← 当前活跃 change
}

加上 WORKFLOW.md <state-schema> 声明的扩展字段（skill_history、external_refs、legacy_source 等）。

change 的生命周期

创建进入协议 Phase 1
原子创建 change 目录
更新 status.json
→
活跃技能产物写入
change/{路由子目录}/
可跨路由复用
→
完成门禁用户确认交付边界
finalize 命令触发
completion-gate 检查
→
归档change 移至 archive/
status.json 更新

5. 数据流向：一次完整的请求追踪

以刚才我们的 teach 请求为例，追踪完整的调用链和数据流向：

1
runtime-context 激活

从 CWD 向上查找 speculo/.speculo/workspace.json，解析根别名：

  • workflow = speculo/workflows
  • state = speculo/.speculo
  • vendor = speculo/vendor

结合 WORKFLOW.md 的 <runtime-context>，推导：

  • workflow_root = workflows/matt-pocock
  • state_root = .speculo/matt-pocock
  • vendor_root = vendor/matt-pocock
2
select-change（Phase 1）

读取 state_root/status.json。若无当前 change，原子创建新 change：

  • 生成 change ID：teach-20260711-190000
  • 创建目录：state_root/changes/teach-20260711-190000/
  • 更新 status.json：current_change = "teach-20260711-190000"

派生 change_root = state_root/changes/teach-20260711-190000

3
route（Phase 2）— ask-matt 激活

读取 vendor_root/engineering/ask-matt/SKILL.md

用户意图「想学 workflows 的使用方法」→ 匹配 productivity 路由 → teach 子路由

更新 status.json：current_route = "productivity"，route_history 追加

4
lazy-config（Phase 3）

teach 子路由不需要 issue tracker / triage labels / domain 文档 → 跳过

5
技能激活 — teach

读取 productivity 路由中的 teach 声明：

  • skill = vendor_root/productivity/teach/SKILL.md
  • activation = adapted → 路径适配模式
  • artifact root = change_root/teaching/

原 skill 写「当前目录」→ 适配为 change_root/teaching/

原 skill 写「./lessons/」→ 适配为 change_root/teaching/lessons/

MISSION.md、RESOURCES.md、lessons/*.html 全部写入适配后的路径

6
持久化与状态转移

每次状态变更更新 status.json（current_route、route_history、skill_history）

用户说「完成」→ finalize 命令 → completion-gate 检查 → change 移至 archive/

6. 层间契约总结

契约上游（调用方）下游（被调用方）核心约束
路径解析契约 workflow 的 <runtime-context> workspace.json + runtime-context skill 所有路径必须是项目根相对路径；任一路径越界或目标缺失 → blocked
路由适配契约 route 的 <skill activation="adapted"> vendor SKILL.md 仅替换路径引用（当前目录、.scratch/、docs/agents/、系统临时目录），不修改技能逻辑
持久化契约 workflow 的 <persistence> state namespace 的目录/文件 固定骨架（status.json/changes/archive）必须存在；懒 store 按需创建；消费者标记决定哪些命令读取哪些 store
状态转移契约 workflow 的 <transitions> status.json route→phase 在路由选定且配置就绪后触发；phase→route 在当前能力完成后触发；active→archived 需 finalize 命令 + completion-gate

7. 自检点

Q1: 如果 vendor skill 写「把文件保存到当前目录」，workflow 层如何确保文件实际写入 state namespace？

提示：看 productivity 路由中 teach 的 activation="adapted" 和 <artifact root="change" path="teaching/">

点击查看答案

Workflow 层在激活技能前进行路径适配：将技能原文中的「当前目录」替换为 <artifact> 声明的路径（如 change_root/teaching/）。这是 activation="adapted" 模式的行为——activation="required" 模式不做路径替换。

Q2: status.json 中的 current_change 和 active 数组是什么关系？

点击查看答案

active 是所有活跃 change ID 的列表（可能多个），current_change 是当前正在使用的 change ID。一个 change 可以跨越多个路由（route→phase→route 转移到新路由时复用同一 change），所以 current_change 可能不变但 current_route 会变。

Q3: 为什么 persist 声明中 knowledge store 的 create="lazy" 而 index store 的 create="initialize"？

点击查看答案

create="initialize" 意味着进入协议阶段必须确保该 store 存在——index（status.json）、changes、archive 是工作流运行的固定骨架，缺少任何一个工作流都无法正确运转。create="lazy" 意味着 knowledge 目录只在某条路由实际产生领域知识时才创建——如果用户只使用不需要领域知识的路由（如 teach），就不需要创建空目录。

1. 进入协议的定位

进入协议是整个工作流的启动序列。无论用户触发哪条路由，工作流总是先执行这三个阶段，确保上下文就绪后才激活目标技能。可以将其理解为操作系统的 bootloader——在加载内核之前，必须先初始化硬件、设置页表、建立内存映射。

每个 phase 有完成标准（completion）——必须满足标准才能进入下一个 phase。Phase 3 还有执行条件（when）——不满足条件则跳过。

2. Phase 1: select-change — 建立上下文

执行流程

1
激活 runtime-context skill

从 CWD 向上遍历，定位 speculo/.speculo/workspace.json

若找不到唯一项目根 → blocked（返回错误，不继续）

2
解析根注册表

读取 workspace.json 的 roots 对象 → 6 个根路径

3
解析 WORKFLOW.md 的 <runtime-context>

将 workflow/state/vendor 三个根别名映射到物理路径

路径越界或目标不存在 → blocked

4
读取 status.json

若 status.json 存在：读取 current_change，派生 change_root

5
原子创建（若需要）

若 status.json 不存在或 current_change 为空：

  • 生成 change ID（格式：{route}-{YYYYMMDD}-{HHmmss}）
  • 创建 changes/{change_id}/ 目录
  • 写入 status.json（含 current_change）

整个过程是原子的——不存在半创建状态

6
返回 runtime context

输出：project_root, workflow_root, state_root, changes_root, archive_root, change_root, commands_root, skills_root, vendor_roots[]

不创建任何运行时产物（status.json 和 change 目录是持久化骨架，不是临时产物）

关键设计决策

  • 为什么 runtime-context 不把派生绝对路径写回 status.json？ 因为 change_root 是派生值（= changes_root / current_change），不是独立状态。写回去会造成数据冗余和一致性问题。
  • 为什么找不到唯一项目根时是 blocked 而非降级？ 工作流必须知道自己在哪个项目、用哪个根注册表。降级意味着路径可能错误，工作流的持久化承诺无法保证。
  • activation="required" 的含义：runtime-context 不可跳过、不可适配。它是工作流的基础设施——如果它可以被跳过，整个工作流的路径上下文就无处获取。

3. Phase 2: route — 匹配意图

ask-matt skill 的内部逻辑

ask-matt 不是一个简单的"关键词匹配"路由器，它是一张技能地图。它的 SKILL.md（vendor/matt-pocock/engineering/ask-matt/SKILL.md）描述了完整的技能生态：

主流程idea-to-delivery
grill → spec → tickets
→ implement
←
入口匝道triage
diagnosing-bugs
wayfinder
←
独立技能teach
grill-me
prototype
research
handoff

路由匹配策略

  1. 读取所有 route 的 <when>：ask-matt 获得完整的条件列表
  2. 按 order 顺序尝试匹配：从 order=1 开始，检查用户意图是否落入该 when 描述的语义范围
  3. 歧义消解：当多个路由匹配时，"一次只澄清一个决策"——问一个能排除最多干扰的问题

    例子：用户说「帮我看看这个 PR」——可能是 review（审查 diff）也可能是 triage（分类外部 PR）。ask-matt 的问法：「这个 PR 是你自己写的需要审查，还是别人提交的需要分类？」——一个问题区分两条路由。

  4. 匹配后更新 status.json：current_route = 匹配到的路由 ID，追加 route_history

route_history 的价值

记录用户走过的路由序列，可用于：

  • 回溯：从 triage 到 implement 到 review，你经过了哪些路由？
  • 分析：哪些路由组合最常见？（route_history 可被 retro 命令消费）
  • 恢复：会话中断后，知道用户上次在哪个路由，可以从那里继续

4. Phase 3: lazy-config — 按需补齐

什么时候触发

仅当目标路由的运行时依赖缺失时才进入此阶段。依赖分为三类：

依赖类型需要的文件依赖它的路由
Tracker 配置integrations/issue-tracker.mdtriage、wayfinder、to-spec、to-tickets、implement
Triage 标签integrations/triage-labels.mdtriage、wayfinder
领域文档knowledge/domain.mdgrill-with-docs、domain-modeling、codebase-design

teach、handoff、code-review 等路由不需要上述任何配置，所以它们的 lazy-config 阶段直接跳过。

setup.md 的四个配置阶段

1
discover-config

激活 setup-matt-pocock-skills 进行现状扫描。展示仓库结构、tracker 现状、领域文档现状。

2
configure-tracker（条件）

仅当路由需要 issue tracker。从 vendor 模板中选择 GitHub/GitLab/local，写入 integrations/issue-tracker.md。

3
configure-triage（条件）

仅当路由为 triage 或 wayfinder。配置五个分类角色（triage/new/confirmed/in-progress/resolved）对应的真实标签。

4
configure-domain（条件）

仅当路由会读取或维护领域语言。创建 knowledge/domain.md，指向本 workflow 的 knowledge/context/ 与 knowledge/adr/。

⚠️ 关键设计点：lazy-config 不是完整运行 setup-matt-pocock-skills。它只补齐目标路由必需的配置。这意味着如果用户先触发 teach（不需要配置），再触发 triage（需要 tracker + labels），lazy-config 会在第二次触发时补齐前两个。这是"渐进配置"模式——从不一次性要求用户配完所有东西。

5. 三步之间的状态流转

用户发言任何意图表达
→
Phase 1runtime-context
→ 有路径上下文
→
Phase 2ask-matt
→ 有路由 + change
→
Phase 3按需补齐配置
→ namespace 就绪
→
技能执行route 中的 skill
开始工作

任何一个 phase 的完成标准未满足，流转不会继续。Phase 1 blocked 意味着整个工作流不可用——这是最早且最硬的失败点。

6. 常见修改场景

场景 A：增加 Phase 4 — 安全检查

<phase id="security-check" order="4">
  <skill root="skills" path="security-audit/SKILL.md" activation="required" />
  <when>目标 route 涉及外部副作用（triage、implement、to-spec）。</when>
  <completion>已确认所有待执行的外部操作安全且用户已确认。</completion>
</phase>

注意 when 条件：安全检查不需要在所有路由上运行（teach 不产生外部副作用），只在涉及 issue 发布、代码提交等路由上运行。

场景 B：将 lazy-config 的条件扩展为路由白名单

当前 when 用的是自然语言，如果你想让其更精确，可以改为路由列表：

<when>
  target_route 属于 [triage, wayfinder, to-spec, to-tickets,
                    implement, grill-with-docs, domain-modeling,
                    codebase-design, improve-codebase-architecture]
</when>

场景 C：让 select-change 支持指定已有 change

当前 select-change 总是创建新 change 或使用当前 change。如果用户想继续一个已归档的 change，可以在进入协议前加一个选择步骤，让用户从 active 列表中选择。

7. 自检点

Q1: 如果 Phase 1 成功但 Phase 2 中 ask-matt 无法匹配任何路由，会发生什么？

点击查看答案

ask-matt 应返回一个"未匹配"状态，向用户说明无法理解意图，并建议更清晰地描述。workflow 回到等待用户输入的状态（类似 route 选定前的状态）。这不同于 blocked——blocked 是基础设施问题，未匹配是语义问题。

Q2: 为什么 lazy-config 用 <instructions> 标签而非 <skill> 标签加载 setup.md？

点击查看答案

<skill> 表示激活一个独立的 vendor 技能（有 SKILL.md 的完整技能目录），<instructions> 表示加载工作流自身的指引文件。setup.md 位于 workflows/matt-pocock/routes/ 下，是工作流的一部分而非独立技能——它存在的唯一目的就是为进入协议提供配置指引。

Q3: 假设你想让某些路由跳过进入协议直接执行技能，可行吗？

点击查看答案

不可行。进入协议是整个工作流的基础设施——如果没有 Phase 1 就没有路径上下文，技能不知道把产物写到哪里；如果没有 Phase 2 就无法确定激活哪个技能。即使是最简单的路由（如 productivity → teach），也必须有 change_root 来作为教学产物的存储位置。跳过进入协议等于跳过操作系统的 bootloader。

路径适配机制

activation="adapted" 如何将 vendor skill 的默认路径重定向到 state namespace

本课内容

  1. 为什么需要路径适配
  2. activation="required" vs "adapted"
  3. 四条重定向规则
  4. 完整示例：teach skill 的路径变换
  5. artifact 路径模板变量

1. 为什么需要路径适配

vendor 技能是独立编写的——它们不知道 Speculo 的存在。一个典型的 vendor skill 会写：

将当前目录视为教学工作区。
- `MISSION.md`：记录用户对该主题感兴趣原因的文档。
- `./lessons/*.html`：课程目录。
- `./assets/*`：跨课程共享的可复用组件。

在 Speculo 中，"当前目录"是项目根——如果 teach 在那里写 MISSION.md 和 lessons/，会污染项目源码。路径适配的本质是：保持技能逻辑不变，但将所有文件系统写入重定向到 state namespace。

2. activation 的两种模式

requiredadapted
适用对象基础设施技能（runtime-context、ask-matt）业务技能（teach、triage、handoff 等）
路径处理保持原样，不做任何重定向将默认路径映射到 artifact 声明的 state namespace 路径
可否跳过不可跳过——缺失则工作流 blocked不可跳过执行，但 lazy-config 阶段可能被跳过
典型场景工作流引擎本身——它需要真实的文件系统路径来构建运行时上下文用户面向的技能——它产生业务产物，需要隔离在 state namespace 中
如果误用对业务技能用 required → 产物写入项目根，污染源码对基础设施用 adapted → 引擎找不到真实路径，工作流失败

为什么 runtime-context 必须是 required？

runtime-context 的工作是找到真实的文件系统路径并验证它们存在。如果对它做路径适配，它会返回适配后的路径（例如把 workspace.json 的路径指向 change/ 内），后续所有技能都无法找到真实的配置文件。

为什么 teach 必须是 adapted？

teach 的工作是产生教学内容。这些内容属于某个 change 的工作产物，不属于项目源码。adapted 模式确保 MISSION.md 写入 change/teaching/MISSION.md 而非 ./MISSION.md。

3. 四条重定向规则

原始路径适配后路径示例
当前目录 (./) <artifact root="change" path="..."> 声明的路径 ./MISSION.md → change/teaching/MISSION.md
.scratch/ change 内的等价临时目录 .scratch/draft.md → change/scratch/draft.md
docs/agents/ change 下的对应文档路径 docs/agents/brief.md → change/docs/brief.md
系统临时目录 (/tmp) change 内的 tmp/ 目录 /tmp/prototype.py → change/tmp/prototype.py

四条规则的优先级：先匹配 artifact 声明，artifact 未覆盖的路径按上述映射处理。如果技能使用了不在上述列表中的路径（如 ~/.config/），则保持原样——这通常是一个 bug，需要在路由层面处理。

⚠️ 路径适配不递归：只适配技能直接指定的路径，不对技能读取的外部文件内容中的路径再做替换。如果技能引用了一个模板文件，模板文件内的路径不会被适配。

4. 完整示例：teach skill 的路径变换

路由声明（productivity.md）

<route id="teach" order="2">
  <skill root="vendor:matt-pocock" path="productivity/teach/SKILL.md"
         activation="adapted" />
  <artifact root="change" path="teaching/" />
  <when>用户要跨会话学习主题。</when>
</route>

变换映射表

技能原文适配后路径说明
MISSION.mdchange/teaching/MISSION.md当前目录 → artifact 路径
./reference/*.htmlchange/teaching/reference/*.html相对路径被保留在 artifact 子目录下
RESOURCES.mdchange/teaching/RESOURCES.md同上
./learning-records/*.mdchange/teaching/learning-records/*.md同上
./lessons/*.htmlchange/teaching/lessons/*.html同上
./assets/*change/teaching/assets/*同上
NOTES.mdchange/teaching/NOTES.md同上

技能本身完全不知道这些文件实际写入 .speculo/matt-pocock/changes/teach-xxx/teaching/——它以为自己在操作"当前目录"。这个隔离性是 adapted 模式的核心价值。

5. artifact 路径模板变量

artifact 的 path 支持模板变量：

<artifact root="change" path="handoffs/{topic}.md" />

{topic} 是运行时变量，由技能在执行时填充。这允许同一路由产生不同名称的产物文件，但仍在声明的路径空间内。

变量来源示例
{topic}用户输入或技能提取的主题名handoffs/prototype-result.md
{date}当前日期（YYYY-MM-DD）weekly-summary/2026-07-11.md
{change_id}当前 change ID可用于产物标记来源 change

模板变量的安全边界：变量值会被 sanitize（去除路径分隔符和特殊字符），防止路径穿越攻击（如 topic = "../../../etc/passwd"）。

6. 自检点

Q1: 如果一个 vendor skill 写了 ~/.config/my-skill/settings.json，adapted 模式会怎么处理？

点击查看答案

保持原样，不做适配。四条重定向规则不覆盖 home 目录路径。这是一个需要注意的场景——技能将数据写到 state namespace 之外，跨 session 可能冲突。最佳实践是避免技能使用 ~ 路径；如果必须，在路由声明中扩展 adapted 规则或在代码审查时发现并修正。

Q2: 如果同一个路由中既要调用一个 adapted skill 又要调用一个 required skill，artifact 如何声明？

点击查看答案

每个 <skill> 有独立的 activation 属性。可以这样设计：

  <skill root="vendor:x" path="business-skill" activation="adapted" />
  <skill root="skills" path="infra-check" activation="required" />
  <artifact root="change" path="output/" />

artifact 声明应用于 adapted 技能；required 技能忽略它。

Q3: 路径适配是编译时（路由加载时）还是运行时（技能写入文件时）发生的？

点击查看答案

逻辑上是运行时——在技能执行过程中，当技能试图写入文件时，拦截写入路径并做替换。实际上由 Speculo 的 workflow 引擎在激活技能时注入路径上下文，技能通过该上下文解析最终路径。不是静态替换技能文件中的文本——技能 SKILL.md 本身保持不变。

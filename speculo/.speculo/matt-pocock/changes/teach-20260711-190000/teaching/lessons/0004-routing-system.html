优先级（order）的含义：数字越小优先级越高。当用户意图模糊时，ask-matt 从高优先级路由开始匹配。这意味着 idea-to-delivery（最宽泛的"我有一个想法"）虽然覆盖的面最广，但因为优先级最高，它总是第一个被尝试。

2. 主流程路由：idea-to-delivery

这是最核心的路线，涵盖了从想法到交付的完整生命周期。它内部是一个序列（to-spec → to-tickets → implement），而非并行。

路由文件结构（routes/idea-to-delivery.md）

与子路由模式（如 productivity 内的 teach/handoff/writing）不同，idea-to-delivery 是线性阶段：

  1. grill-with-docs（可选但有状态）——通过访谈打磨想法，将领域知识写入 CONTEXT.md 和 ADR
  2. to-spec——将对话综合为规范
  3. to-tickets——将规范拆分为 tracer-bullet 工单
  4. implement——逐个工单构建（内部驱动 tdd + code-review）

idea-to-delivery 路由与 ask-matt 中描述的"主流程"完全对应。路由文件负责声明在 Speculo 框架中如何执行这些步骤（路径、持久化、状态转移），而 ask-matt 负责描述在 Matt 方法论中何时选择这条路线。

3. 入口匝道路由

triage（order=3）

专门处理外部到达的问题——bug 报告、功能请求、外部 PR。核心是五角色状态机：

new未读
→
triage分类中
→
confirmed已确认
→
in-progress处理中
→
resolved已解决

产物包括分类决策、标签更新、生成的 agent 可处理工单。完成 triage 后，通常转移到 idea-to-delivery 路由进行实现。

diagnose（order=4）

规范化诊断循环：重现 → 最小化 → 假设 → 插桩 → 修复 → 回归测试。关键约束：在拥有紧凑反馈回路（一条在此 bug 上变红的命令）之前拒绝推测。如果真正的问题是缺少好的缝合点来锁定 bug，事后分析移交给 architecture 路由。

wayfinder（order=2）

优先级仅次于 idea-to-delivery，因为它是入口的最前端——当用户甚至不清楚路径是否存在时。核心产出是 issue tracker 上的调查工单地图，每个工单是一个待解决的决策问题。解决完所有调查工单后，若工作量很大 → to-spec；若工作量很小 → implement。

4. 代码仓健康路由

architecture（order=5）

扫描代码库寻找深化机会（deepening opportunities），以可视化 HTML 报告呈现。报告是普查——列出所有可能的改进方向。用户从中选择一个后，生成一个想法，带入主流程的 grill-with-docs 进行打磨。architecture 路由是"找到候选方案"的阶段——不设计解决方案，只发现问题。

review（order=6）

从固定点开始的双轴审查：

  • 标准轴：是否遵循仓库的编码标准 + Fowler 气味基线？
  • 规范轴：是否忠实实现了原始 issue/PRD？

两轴作为并行子 agent 运行，互不污染。

merge-conflicts（order=7）

仅在仓库处于冲突状态时触发。这是一个条件路由——git status 显示冲突时自动提升优先级。

5. 独立路由

research-prototype（order=8）

两个互补能力的组合：

  • research：后台 agent 对照一手资料调查问题，留下带引用的 Markdown
  • prototype：一次性程序回答单个设计问题（状态模型？UI 变体？）——保留答案，删除代码

两者都可以在主流程的 grill 阶段被调用，也可以独立使用。

productivity（order=9）

三个子路由（handoff / teach / writing-great-skills），全部独立于编码主流程，属于元工具：

  • handoff：会话间上下文传递的桥梁
  • teach：跨会话的有状态教学（你现在正在使用的）
  • writing-great-skills：skill 编写参考指南

experimental（order=10）

最低优先级路由，仅在用户明确要求实验能力时触发。指向 vendor/in-progress/ 中的未成熟技能。这些技能可能改变或移除，不应在其他路由中依赖。

6. 路由间转移

路由不是孤岛——它们通过 WORKFLOW.md 的 <transitions> 连接：

→

→ implement

→
idea-to-deliveryto-spec

所有路由最终汇聚到 idea-to-delivery，因为它是唯一一条负责"构建和交付"的路由。其他路由都是为这条主流程做准备——发现问题、分类问题、规划路径、打磨想法。

转移时的 change 处理

当路由间转移时，同一 change 被复用。例如从 triage 转移到 implement 时，current_change 不变，但 current_route 更新。这意味着一个 change 目录可以包含多个路由的产物（如 triage 报告 + spec + 实现提交记录），形成一个完整的"从发现问题到解决问题"的 trace。

7. 实操：新增一条路由的完整步骤

假设你要新增路由 "weekly-summary"——自动生成周报。

需要修改的部分（按依赖顺序）

1
确认 vendor 技能存在

是否有现成的 vendor skill 可以生成周报？如果没有，需要先在 vendor 中添加（或使用已有技能的变体）。假设我们复用 research 技能来收集本周的提交记录和 issue 变更。

2
创建路由文件

workflows/matt-pocock/routes/weekly-summary.md

<routes>
  <route id="generate-weekly" order="1">
    <skill root="vendor:matt-pocock"
           path="engineering/research/SKILL.md"
           activation="adapted" />
    <artifact root="change" path="weekly-summary/{date}.md" />
    <when>用户要生成周报或本周工作总结。</when>
  </route>
</routes>
3
在 WORKFLOW.md 的 <routes> 中添加条目 
<route id="weekly-summary" order="11"
       root="workflow" path="routes/weekly-summary.md">
  <when>用户要生成周报、本周总结或工作回顾。</when>
</route>

order=11 放在 experimental 之后（最低优先级），或者根据重要性调整。

4
评估是否需要新 store

周报产物在 change/weekly-summary/ 下，使用已有的 changes store。不需要新增 persistence store。但如果周报需要跨 change 的汇总数据（如「本月所有周报的聚合」），可能需要一个 backlog store 或在 knowledge 中新增。

5
评估是否需要扩展 state-schema

如果周报需要追踪「上次生成日期」或「已生成周报列表」，在 state-schema 中新增字段。

可以看出：新增一条不需要新存储和不需要新状态字段的简单路由，只需要修改 2 个文件（WORKFLOW.md 的 routes 块 + 新路由文件）。核心复杂度在 vendor 技能本身，编排层的修改量很小。

8. 自检点

Q1: 为什么 idea-to-delivery 优先级最高（order=1）而 experimental 最低（order=10）？

点击查看答案

idea-to-delivery 覆盖范围最广（任何"推进想法"的意图都可能匹配），如果不放在最高优先级，它可能被更窄的路由劫持。experimental 是实验性质的、不稳定的，放在最低优先级确保它只在用户明确要求时触发，不会被意外匹配。优先级的本质是：宽泛但核心的路由排前面，窄但危险的路由排后面。

Q2: triage 的产物包括"agent 可处理工单"。这些工单存在哪里？当转移到 implement 时如何被找到？

点击查看答案

工单存在当前 change 的 tracker 子目录或 issue tracker 集成中。triage 完成后，status.json 中有 external_refs 记录工单的引用。当用户说「实现这个工单」时，ask-matt 检测到有现有工单且意图是实现 → 匹配 idea-to-delivery 路由，implement 技能读取同一 change 下的 tracker 找到待实现工单。关键：same change = same trace，产物在同一个 change 目录下自然可见。

Q3: 如果用户说「帮我审查代码并生成周报」，ask-matt 应该匹配哪条路由？

点击查看答案

两个意图：审查（review，order=6）和生成周报（我们假设的 weekly-summary，order=11）。按照进入协议"一次只澄清一个决策"的原则，ask-matt 会先问：「你是想先审查代码，还是先生成周报？」或者更精准：「你当前主要想做什么——审查还是总结？」——处理完第一个后，进入 phase→route 转移，再处理第二个。

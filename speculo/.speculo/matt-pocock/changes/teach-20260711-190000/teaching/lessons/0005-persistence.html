1. status.json 的完整字段说明

骨架字段（所有工作流通用，无需在 state-schema 中声明）

字段类型语义谁写入
schema_versionnumber状态格式版本号。升级时用于数据迁移工作流初始化
workflowstring所属工作流 ID，对应 WORKFLOW.md 的 frontmatter.id工作流初始化
current_routestring当前活跃路由 IDPhase 2 (route)
route_historyarray历史路由 ID 序列，按时间顺序追加每次路由变更时追加
activearray所有活跃（未归档）change ID 的列表select-change 创建时追加，finalize 归档时移除
current_changestring当前正在使用的 change IDselect-change

扩展字段（在 state-schema 中声明）

字段类型语义
skill_historyarray调用的技能名称序列，用于回溯工作流历史
external_refsarray外部系统引用的对象数组：{type, url, created_at, change}

⚠️ 字段不是 schema-less 的：state-schema 中的 type 声明目前是文档性质（告诉维护者这个字段是什么类型），但在未来的 Speculo 版本中可能变为运行时校验。修改字段名或类型时需要考虑向后兼容。

2. change 生命周期：从创建到归档

1
创建 — select-change

Phase 1 发现无当前 change → 生成 change ID（格式：{route}-{YYYYMMDD}-{HHmmss}）→ 原子创建目录和 status.json

change 目录物理路径：.speculo/matt-pocock/changes/{change_id}/

此时 change 目录为空——产物由后续技能写入

2
活跃 — 技能写入产物

每个路由的 <artifact> 声明产物路径。例如 teach 路由：artifact root="change" path="teaching/" → 产物写入 change/teaching/

一条 change 可以跨多个路由：triage → 分类报告写入 change/triage/，然后 phase→route 转移到 implement，实现产物写入 change/implement/。同一 change 的不同子目录记录了完整的工作流 trace。

3
完成门禁 — finalize

用户说「完成」或「这个 change 做完了」→ 触发 finalize 命令 → completion-gate 检查（见下节）→ 通过后进入归档

若不通过，命令返回具体的不通过原因（如「有未解决的 triage 工单」「有未提交的代码」），change 仍保持活跃

4
归档 — archive/

change 目录从 changes/ 移动到 archive/

status.json 更新：从 active 数组中移除，current_change 清空或指向下一个活跃 change

归档后 change 变为只读（不应再修改），但可供 retro 命令读取做回顾分析

多 change 共存

active 数组允许多个 change 同时活跃。例如：

  • 你正在 triage 一个新 issue（change-A）
  • 同时你需要 implement 一个之前已分好工单的任务（change-B）
  • 两个 change 都是活跃的，但同一时刻只有一个 current_change
  • 通过某种切换机制（未来功能），可以在 change 之间切换

3. finalize 命令与 completion-gate

finalize 是唯一从 active→archived 的转移通道。它不同于自动转移（route→phase 和 phase→route 是自动的），需要用户显式触发 + 门禁检查通过。

completion-gate 检查清单（由 skill change-lifecycle 定义）

检查项不通过时的行为
当前路由的所有 phase 是否已完成？列出未完成的 phase，要求用户先完成或明确放弃
是否有未提交的代码变更？列出变更，询问是否提交、放弃或保留在 change 外
是否有未解决的 external_refs？列出外部引用，要求确认状态（已发布/已关闭/已合并）
是否有临时原型代码残留？列出原型文件，要求删除（保留结论）
change 目录是否包含所有声明的产物？列出缺失的产物

设计原则：finalize 不做自动清理——它暴露所有未完成项让用户做决策。这保证了用户对交付边界有完全的控制。一个 change 只有在用户确认"这个交付单元已经完整且干净"后才会归档。

4. external_refs：审计追踪

当技能产生不可逆的外部副作用时（发布 issue、添加标签、提交代码、创建 PR、合并分支），在操作前：

  1. 展示即将执行的动作给用户看
  2. 等待用户明确确认
  3. 执行后将结果写入 external_refs

external_refs 条目格式（推断）

{
  "type": "github-issue",        // 外部系统类型
  "url": "https://github.com/...", // 外部资源定位
  "action": "created",            // 执行的动作
  "change": "triage-20260711-xxx", // 关联的 change
  "confirmed_at": "2026-07-11T19:30:00Z"  // 用户确认时间
}

external_refs 的价值：

  • 审计：回顾时知道这个 change 触发了哪些外部操作
  • 恢复：会话中断后知道哪些操作已完成，无需重复执行
  • 回滚：如果需要撤销（例如发布了错误的 issue），有完整的引用列表

5. legacy-config：旧格式兼容

<store id="legacy-config" role="legacy-knowledge" kind="directory"
       path=".config" create="existing-only" legacy="true"
       consumers="docs-sync,retro,knowledge-prune" />
  • create="existing-only"：如果 .config/ 目录存在就读取，不存在就静默跳过。不会为了兼容而创建空目录
  • legacy="true"：标记为旧格式。consumers 从中读取数据但不会写入新数据
  • consumers：docs-sync、retro、knowledge-prune 三个命令会读取 legacy-config 中的旧数据，用于迁移和兼容

这种设计允许 Speculo 的存储格式升级而不断裂——旧格式的数据仍然可读，新数据写入新的 store。当所有 consuming 命令都不再需要旧格式时，可以安全移除 legacy-config store。

6. 四条持久化与副作用规则

①
产物写入当前 change

所有本地记录 → 当前 change。长期知识/规则/集成配置/backlog → 声明的 state namespace。

反向例子（不允许）：teach 技能把 MISSION.md 写到项目根目录、triage 把工单写进 /tmp。

②
路径重定向

原 skill 指向的四个默认路径全部被替换：

  • 项目根目录 → change/ 对应子目录
  • .scratch/ → change/scratch/
  • docs/agents/ → change/ 下对应路径
  • 系统临时目录 → change/tmp/
③
外部副作用确认

以下操作必须展示 + 确认：发布 issue、评论、标签变更、代码提交、分支合并、worktree 删除。结果写入 external_refs。

④
临时产物清理

项目代码/测试/文档 ≠ Speculo 产物。原型代码必须在得出结论后删除——保留结论（如设计决策记录），删除代码。

7. 自检点

Q1: active 数组允许多个 change 同时存在，但什么情况下会有多个活跃 change？

点击查看答案

(1) 用户中断了某个 change 的工作，开始了一个新任务——旧 change 尚未完成，仍是活跃的。(2) 通过 handoff 分叉——原始会话产生一个 change，prototype 会话产生另一个 change，两者都活跃但属于不同上下文。(3) 并行工作——例如 research 后台 agent 产生一个 change，用户同时在进行 triage 产生另一个 change。

Q2: 如果不经过 finalize 直接手动移动 change 到 archive，会有什么风险？

点击查看答案

status.json 不会更新（active 数组仍包含已移动的 change），completion-gate 检查被绕过（可能有未提交代码、未清理原型、未完成的 phase），external_refs 可能不完整。工作流状态与实际文件系统状态不一致——这是最危险的错误模式。

Q3: legacy-config 的 consumers 字段写了三个命令。如果新增一个命令也需要读取旧配置，如何修改？

点击查看答案

在 WORKFLOW.md 的 persistence 块中，修改 legacy-config store 的 consumers 属性，追加新命令名：consumers="docs-sync,retro,knowledge-prune,new-command"。不需要修改其他文件。consumers 字段是声明式的——它告诉 Speculo 哪些命令有权读取该 store。

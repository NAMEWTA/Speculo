1. 总览：WORKFLOW.md 的结构

WORKFLOW.md 是一份混合文档——YAML frontmatter 声明元数据，正文中嵌入 XML 块描述工作流语义。它的设计原则是：每个 XML 块是一个独立的领域声明，块之间通过根别名、store id、route id、field name 等标识符相互引用。

---
id: matt-pocock
type: workflow
workflow: matt-pocock
name: Matt Pocock Workflow
description: 以路由优先方式组合原生 Matt Pocock skills
keywords: [matt-pocock, engineering, productivity, grilling, tdd, review]
---                  ← YAML 元数据（供 Speculo 注册表索引）

<runtime-context>    ← XML 块 1：路径别名
<persistence>         ← XML 块 2：存储声明
<routes>              ← XML 块 3：路由表
<sequence>            ← XML 块 4（进入协议）：select-change
<sequence>            ← XML 块 5（进入协议）：route
<sequence>            ← XML 块 6（进入协议）：lazy-config
<dependencies>        ← XML 块 7：前置依赖
<state-schema>        ← XML 块 8：状态扩展字段
<transitions>         ← XML 块 9：状态转移机
[正文段落]             ← 自然语言规则：持久化与副作用

下面逐块深入，不只看「写了什么」，还要看「为什么这样写」和「修改时要注意什么」。

2. <runtime-context> — 根别名映射

<runtime-context>
  <root id="workflow" base="workflows" path="matt-pocock" />
  <root id="state" base="state" path="matt-pocock" />
  <root id="vendor:matt-pocock" base="vendor" path="matt-pocock" />
</runtime-context>

每个属性的语义

属性含义解析方式
id根别名，后续所有 XML 块和路由文件中用此名称引用该根字符串匹配。命名空间用 : 分隔（vendor:matt-pocock）
base指向 workspace.json 中 roots 对象的键名先查 workspace.json 的 roots.{base} 得到物理路径，再拼接 path
path在 base 解析后的物理路径下的子路径与 base 解析结果拼接，形成完整相对路径

解析链路追踪

1
runtime-context skill 读取 workspace.json

roots.workflows = "speculo/workflows"

2
解析 root id="workflow"

base="workflows" → speculo/workflows

path="matt-pocock" → speculo/workflows/matt-pocock

3
解析 root id="state"

base="state" → workspace.json: roots.state = "speculo/.speculo"

path="matt-pocock" → speculo/.speculo/matt-pocock

4
解析 root id="vendor:matt-pocock"

base="vendor" → workspace.json: roots.vendor = "speculo/vendor"

path="matt-pocock" → speculo/vendor/matt-pocock

⚠️ 修改注意事项：id 名称的变更会影响所有引用该根的地方。例如将 vendor:matt-pocock 改为 vendor:mp，需要同步修改所有路由文件和进入协议中的 root="vendor:matt-pocock"。base 的值必须与 workspace.json 中的键名完全一致，path 的值必须指向真实存在的目录。

3. <persistence> — 持久化命名空间

<persistence root="state">
  <store id="index" role="index" kind="file" path="status.json"
         create="initialize" />
  <store id="changes" role="active" kind="directory"
         path="changes" create="initialize" />
  <store id="archive" role="archive" kind="directory"
         path="archive" create="initialize" />
  <store id="knowledge" role="knowledge" kind="directory"
         path="knowledge" create="lazy"
         consumers="docs-sync,retro,knowledge-prune" />
  <store id="policy" role="policy" kind="directory"
         path="policy" create="lazy"
         consumers="docs-sync,knowledge-prune" />
  <store id="integrations" role="integration" kind="directory"
         path="integrations" create="lazy" />
  <store id="backlog" role="backlog" kind="directory"
         path="backlog" create="lazy" consumers="retro" />
  <store id="legacy-config" role="legacy-knowledge" kind="directory"
         path=".config" create="existing-only" legacy="true"
         consumers="docs-sync,retro,knowledge-prune" />
</persistence>

store 属性详解

属性含义可选值
root 该持久化块的父根，所有 store 的 path 相对于此根解析 必须是 <runtime-context> 中声明的 id
idstore 的逻辑标识符，在路由文件和转移规则中被引用任意字符串
rolestore 的语义角色，供命令和 skill 按角色而非 id 发现 storeindex, active, archive, knowledge, policy, integration, backlog, legacy-knowledge
kind存储的物理形态file（单个文件）或 directory（目录）
path相对于 persistence root 的文件系统路径相对路径字符串
create关键属性：控制 store 的创建时机见下表
consumers声明哪些 Speculo 命令会消费此 store 中的数据逗号分隔的命令名列表
legacy标记为兼容旧格式，不作为新内容默认写入位置true 或不设置

create 的三种模式

值行为适用场景
initialize 工作流启动时必须确保存在，不存在则创建空文件/目录 固定骨架：status.json、changes/、archive/——缺少任何一个工作流无法运转
lazy 仅在路由产生内容时才创建，不预先创建空目录 可选数据：knowledge/、integrations/、backlog/——如果用户只用 teach 路由，不需要创建 knowledge 目录
existing-only 只读取不创建。如果目录不存在，跳过而不报错 兼容旧格式：legacy-config 的 .config/——来自旧版的配置，如果存在就读取迁移，不存在不影响工作流

⚠️ 修改注意事项：添加新 store 时选择合适的 create 模式至关重要。设为 initialize 的 store 会强迫每个工作流实例都创建该目录，容易造成空目录污染。需要新增消费者命令时更新 consumers 字段，否则命令可能找不到它需要的数据。

4. <routes> — 路由表

<routes>
  <route id="idea-to-delivery" order="1"
         root="workflow" path="routes/idea-to-delivery.md">
    <when>用户要把想法、需求或明确任务推进为规范、实现和交付。</when>
  </route>
  <route id="wayfinder" order="2" ...>
  <route id="triage" order="3" ...>
  ...
  <route id="experimental" order="10" ...>
</routes>

路由匹配规则

  1. 优先级由 order 决定：数字越小优先级越高。当用户意图模糊时，ask-matt 从 order=1 开始向下匹配
  2. <when> 是自然语言触发条件：它不是精确的正则匹配，而是给 ask-matt skill 的语义指引——ask-matt 读取所有 route 的 when 描述，判断用户意图最接近哪个
  3. 每条 route 指向一个物理文件：root="workflow" path="routes/xxx.md" → workflows/matt-pocock/routes/xxx.md
  4. 歧义时一次只澄清一个决策：进入协议的注释明确写了「歧义时一次只澄清一个决策」，ask-matt 不会列出所有可能的路由让用户选，而是问一个能排除最多干扰的问题

route 文件内部的子路由模式

以 productivity.md 为例，路由文件内部可以定义进一步的子路由：

<routes>
  <route id="handoff" order="1">
    <skill root="vendor:matt-pocock" path="productivity/handoff/SKILL.md"
           activation="adapted" />
    <artifact root="change" path="handoffs/{topic}.md" />
    <when>需要跨会话或分叉上下文。</when>
  </route>
  <route id="teach" order="2">
    <skill root="vendor:matt-pocock" path="productivity/teach/SKILL.md"
           activation="adapted" />
    <artifact root="change" path="teaching/" />
    <when>用户要跨会话学习主题。</when>
  </route>
</routes>

每个子路由声明三个关键元素：

  • <skill>：目标 vendor 技能的路径和激活模式
  • <artifact>：产物路径模板（支持 {topic} 等变量）
  • <when>：更细粒度的触发条件

5. 进入协议 — <sequence> 三步启动

<sequence>
  <phase id="select-change" order="1">
    <skill root="skills" path="runtime-context/SKILL.md"
           activation="required" />
    <artifact root="state" path="status.json" />
    <completion>已选择唯一 active change，或原子创建新 change。</completion>
  </phase>
  <phase id="route" order="2">
    <skill root="vendor:matt-pocock" path="engineering/ask-matt/SKILL.md"
           activation="required" />
    <completion>已根据用户意图选择一个 route。</completion>
  </phase>
  <phase id="lazy-config" order="3">
    <instructions root="workflow" path="routes/setup.md" />
    <when>目标 route 依赖尚未配置的 tracker、triage 或 domain 文档。</when>
    <completion>只补齐目标 route 必需的私有 namespace。</completion>
  </phase>
</sequence>

Phase 1: select-change

  • 激活 runtime-context skill（activation="required"：必须完整执行，不可跳过）
  • skill 的 root 是 skills，不在 <runtime-context> 中声明——这意味着它直接使用 workspace.json 中 roots.skills 的路径
  • 产物：读写 state/status.json
  • 完成标准：已有唯一活跃 change，或原子创建新 change 并写入 status.json

Phase 2: route

  • 激活 ask-matt skill → 读取所有 route 的 <when> → 匹配用户意图
  • 歧义消解策略："一次只澄清一个决策"——不要列出 5 个候选让用户选，而是问 1 个能排除一半选项的问题

Phase 3: lazy-config

  • 有条件执行：只有当目标路由依赖未配置的资源时才触发
  • 通过 <instructions> 而非 <skill> 加载 setup.md（setup 不是独立技能，而是工作流配置指引）
  • "只补齐必需的"——不是一个完整的 setup-matt-pocock-skills 运行，而是按需配置

⚠️ activation 的两个值：required = 必须完整运行，不可跳过（用于基础设施 skill 如 runtime-context、ask-matt）；adapted = 运行但路径适配（用于业务 skill 如 teach、triage，将其默认路径重定向到 state namespace）。

6. <dependencies> — 前置依赖

<dependencies>
  <dependency kind="hard" root="vendor:matt-pocock" path="README.md">
    选择本 workflow 时必须安装完整 matt-pocock vendor 目录。
  </dependency>
</dependencies>
  • kind="hard"：硬依赖。缺失时工作流不可用（对比 soft 依赖：缺失时可降级运行）
  • 这里的依赖声明是声明式的——它告诉 Speculo 在激活工作流前检查 vendor 目录是否完整
  • 依赖的描述文本既是给人看的文档，也是给 Speculo 的错误消息模板

7. <state-schema> — 状态扩展字段

<state-schema>
  <field name="current_route" type="string" />
  <field name="route_history" type="array" />
  <field name="skill_history" type="array" />
  <field name="external_refs" type="array" />
  <field name="legacy_source" type="object|null" />
</state-schema>

这些是 status.json 中骨架字段之外的扩展字段。骨架字段（schema_version、workflow、current_route、active、current_change）是工作流通用的，不需要声明。扩展字段则是本工作流特定的：

  • skill_history：记录调用过的技能序列，用于回溯调试
  • external_refs：记录外部副作用的引用（如创建的 GitHub issue URL），用于审计
  • legacy_source：从旧 .config 目录迁移数据时的来源标记

8. <transitions> — 状态转移规则

<transitions>
  <transition from="route" to="phase">
    <when>route 已选定且所需 namespace 可用。</when>
  </transition>
  <transition from="phase" to="route">
    <when>当前能力完成并需要进入另一条路线。</when>
  </transition>
  <transition from="active" to="archived">
    <command root="commands" path="finalize.md" />
    <when>用户确认交付边界且完成门禁通过。</when>
  </transition>
</transitions>

三条转移规则的工作流状态机

状态route 选定中
→
转移route→phase
路由已选定
namespace 就绪
→
状态技能执行中
→
转移phase→route
当前能力完成
需进入另一条路线
→
状态回到 route 选定
状态active
change 活跃中
→
转移active→archived
用户确认交付
finalize 命令
completion-gate 通过
→
状态archived
change 已归档

关键细节：route→phase 是自动的——一旦路由选定且 namespace 就绪就触发；phase→route 也是自动的——当前能力完成后若用户提出新意图就触发；但 active→archived 需要显式的人机交互——用户确认 + finalize 命令 + completion-gate 检查。

9. 持久化与副作用规则（正文段落）

WORKFLOW.md 最后一部分是自然语言段落，定义了 4 条硬性规则：

规则含义违反后果
产物写入 change 所有本地记录先写入当前 change；长期知识、规则等只写入声明的 state namespace 产物散落在项目根目录，污染代码仓
路径重定向 原 skill 指向根目录、.scratch/、docs/agents/、系统临时目录的路径，改用当前 change 或声明 namespace 中的等价路径 原始路径不可控，跨会话数据丢失
外部副作用确认 发布 issue、评论、标签、提交代码、合并或删除 worktree 前必须展示动作并获得明确确认 不可逆的外部操作在用户不知情的情况下执行
临时产物清理 项目代码、测试和用户要求更新的项目文档不是 Speculo 运行时产物。临时原型必须删除，保留其结论 原型代码混入项目源文件

核心依赖链：workspace.json → runtime-context（别名）→ 所有其他块（引用别名）→ persistence（store）→ transitions（引用 store 和 command）→ status.json（运行时实例）。

11. 自检点

Q1: 为什么 persistence 中 index store 的 kind 是 file 而 changes 的 kind 是 directory？如果把它们对调会怎样？

点击查看答案

index 对应 status.json（单个文件），changes 对应 changes/ 目录（包含多个 change 子目录）。对调会导致 runtime-context 把 changes/ 当作文件读取失败，或尝试在 status.json 路径下创建子目录。

Q2: activation="required" 和 activation="adapted" 的区别是什么？为什么 select-change 用 required 而 teach 用 adapted？

点击查看答案

required：基础设施技能，必须完整按原样执行，不做路径适配。runtime-context 和 ask-matt 是工作流引擎本身，它们输出的是路径上下文和路由决策——如果适配它们的路径，引擎就无法工作。adapted：业务技能，执行逻辑不变但路径需要重定向到 state namespace。teach 原本写"当前目录"，adapted 模式下替换为 change/teaching/。

Q3: 如果要新增一条路由「当用户说『帮我写周报』时」，需要修改哪些 XML 块和文件？

点击查看答案

(1) 在 <routes> 中添加新 <route>，分配 order 值；(2) 创建 workflows/matt-pocock/routes/weekly-report.md 路由文件，声明 skill/artifact/when；(3) 如果有新的持久化需求，在 <persistence> 中添加新 store；否则复用现有的 change store。不需要修改 <runtime-context>、<state-schema> 和 <transitions>（转移规则对此场景仍然适用）。

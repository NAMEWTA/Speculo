2. 场景 1：新增一条简单路由

需求：新增路由 "weekly-summary"，当用户说「生成周报」时激活研究类技能生成周报。

Step 1：创建路由文件

文件：workflows/matt-pocock/routes/weekly-summary.md

# Weekly Summary

<routes>
  <route id="generate-weekly" order="1">
    <skill root="vendor:matt-pocock"
           path="engineering/research/SKILL.md"
           activation="adapted" />
    <artifact root="change" path="weekly-summary/{date}.md" />
    <when>用户要生成周报、本周总结或工作回顾。</when>
  </route>
</routes>

Step 2：注册路由

在 WORKFLOW.md 的 <routes> 块末尾（order=10 experimental 之后）添加：

<route id="weekly-summary" order="11"
       root="workflow" path="routes/weekly-summary.md">
  <when>用户要生成周报、本周总结或工作回顾。</when>
</route>

Step 3：验证清单

  • ✅ vendor skill 存在（engineering/research/SKILL.md）
  • ✅ 不需要新增 persistence store（产物写入已有 changes）
  • ✅ 不需要扩展 state-schema
  • ✅ 不需要修改进入协议（现有三步足够）
  • ✅ order 值不与现有路由冲突
  • ✅ root 别名在 runtime-context 中已声明

总修改量：1 个新文件 + WORKFLOW.md 中新增 4 行。

3. 场景 2：修改现有路由的 skill 映射

需求：将 productivity 路由中的 handoff 子路由改为使用自定义的 handoff 变体技能。

修改前

<skill root="vendor:matt-pocock" path="productivity/handoff/SKILL.md"
       activation="adapted" />

修改后

<skill root="vendor:matt-pocock" path="productivity/handoff-v2/SKILL.md"
       activation="adapted" />

影响分析

影响范围是否需要额外修改
进入协议不需要——handoff 通过 productivity 路由间接激活
status.json不需要——route_history 记录的是路由 ID 而非 skill 路径
其他路由不需要——各路由独立
artifact 路径可能需要——如果新 skill 的产物结构不同，调整 artifact path

总修改量：routes/productivity.md 中改 1 行。

4. 场景 3：新增一个 persistence store

需求：新增 "templates" store，存放跨路由复用的模板文件。

在 persistence 块中添加

<store id="templates" role="template" kind="directory"
       path="templates" create="lazy"
       consumers="docs-sync" />

决策检查清单

  1. create 应该用哪种模式？→ lazy。模板只在有路由需要时才创建，不应预先创建空目录
  2. 需要 consumers 吗？→ 至少需要一个消费者命令，否则 store 永远不被读取。docs-sync 可能会同步模板文档
  3. role 语义是否正确？→ template 不是预定义的 role，但 role 是自由字符串，只要 consumers 命令知道如何按 role 发现即可
  4. 需要更新 state-schema 吗？→ 不需要——新 store 是目录存储，不影响 status.json 的 JSON 字段

5. 场景 4：调整路由优先级

需求：wayfinder 的优先级应该高于 triage，因为用户在开始任何工作前应该先评估全局路径。

⚠️ 高风险操作。路由优先级影响所有用户的意图匹配结果。调整 order 值可能导致之前匹配到 A 路由的意图现在匹配到 B 路由。

修改步骤

  1. 在 WORKFLOW.md 中交换 wayfinder 和 triage 的 order 值（或重新编号所有 route）
  2. 检查两条路由的 <when> 条件是否会导致歧义
  3. 如果两个 when 条件有重叠，需要在 ask-matt 中增加消歧逻辑
  4. 在测试环境中验证：同样的用户输入，路由结果是否仍正确

order 间隔建议

当前 order 是连续数字（1-10），这意味着如果要插入新路由，需要移动所有后续路由。更好的做法是用 10 的倍数（10, 20, 30...），给未来插入留空间。

6. 场景 5：修改进入协议

需求：在 Phase 2 和 Phase 3 之间插入 Phase 2.5——"intent-confirm"，向用户确认：「你确定要执行 {route_name} 吗？这将创建新的 change。」

<phase id="intent-confirm" order="2.5">
  <when>目标 route 会产生外部副作用或创建新 change。</when>
  <completion>用户已确认意图和即将产生的副作用范围。</completion>
</phase>

影响分析

  • Phase 1 (select-change)：不受影响——change 已在此时创建或选择
  • Phase 2 (route)：不受影响——route 已选定
  • Phase 3 (lazy-config)：order 从 3 变为 4（或保持 3 但 intent-confirm 用 2.5）
  • 新增技能？此 phase 没有 skill 声明——它是纯工作流逻辑（展示路由信息和副作用范围 + 等待确认），不需要单独的 vendor skill

最危险的修改：runtime-context 的根别名——它影响所有引用该别名的 XML 块（persistence 的 root、routes 的 root、进入协议的 skill root）。最安全的修改：route 文件内部（只需确保 skill 路径和 artifact 路径有效）。

8. 自检点

Q1: 如果要新增一个 persistence store 但忘记更新 consumers，会发生什么？

点击查看答案

store 会被创建，数据会被写入，但没有命令知道如何读取它——数据变成孤岛。consumers 字段是命令的"读取许可"——没有声明的命令不会扫描该 store。如果是写入型 store（技能直接写），影响较小（产物仍存在 change 中）；如果是读取型 store（如 knowledge），数据无法被 retro/docs-sync 等命令消费。

Q2: 修改 WORKFLOW.md 的 <routes> 中某条 route 的 order 值时，需要同步修改 route 文件本身吗？

点击查看答案

不需要。route 文件内部的子路由 order 是独立编号空间（只在该路由文件内部排序），与 WORKFLOW.md 主路由的 order 是分开的。修改主路由的 order 只影响主路由之间的优先级；修改子路由的 order 只影响该路由文件内部的子路由优先级。

Q3: 删除一条路由需要修改哪些文件？

点击查看答案

(1) 从 WORKFLOW.md 的 <routes> 中删除对应条目；(2) 可选：删除 routes/xxx.md 路由文件（不删也没影响，它不会被引用）；(3) 检查 status.json 的 route_history 是否有该路由——不需要修改 history，但未来 retro 分析时可能遇到引用不存在的路由；(4) 如果该路由有专用的 persistence store（且无其他路由使用），从 persistence 块中删除。

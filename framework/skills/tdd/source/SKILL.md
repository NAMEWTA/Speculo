# 测试驱动开发

## 理念

**核心原则**：测试应通过公共接口验证行为，而不是实现细节。代码可以完全重写；测试不应该。

**好的测试**是集成式的：它们通过公共 API 运行真实的代码路径。它们描述系统「做什么」，而不是「怎么做」。好的测试读起来像规格说明——「用户可以用有效购物车结算」准确地告诉你存在什么能力。这类测试能经受住重构，因为它们不关心内部结构。

**坏的测试**与实现耦合。它们 mock 内部协作者、测试私有方法，或通过外部手段验证（比如直接查询数据库而不是使用接口）。警告信号：重构时测试坏了，但行为没变。如果你重命名了一个内部函数测试就失败了，那些测试测的是实现，不是行为。

示例参见 [tests.md](tests.md)，mock 指南参见 [mocking.md](mocking.md)。

## 反模式：水平切片

**不要先写全部测试，再写全部实现。** 这是「水平切片」——把 RED 当作「写全部测试」，把 GREEN 当作「写全部代码」。

这会产生**垃圾测试**：

- 批量写的测试测的是「想象中的」行为，不是「实际的」行为
- 你最终测的是「形态」（数据结构、函数签名）而不是面向用户的行为
- 测试对真实变化不敏感——行为坏了测试通过，行为正常测试却失败
- 你在还没理解实现之前就确定了测试结构，等于在黑暗中赶路

**正确做法**：通过追踪弹进行垂直切片。一个测试 → 一个实现 → 重复。每个测试都是基于你在上一轮学到的东西做出的响应。因为你刚刚写了代码，你确切地知道什么行为重要以及如何验证它。

```
错误（水平）：
  RED：   test1、test2、test3、test4、test5
  GREEN：impl1、impl2、impl3、impl4、impl5

正确（垂直）：
  RED→GREEN：test1→impl1
  RED→GREEN：test2→impl2
  RED→GREEN：test3→impl3
  ...
```

## 工作流

### 1. 规划

探索代码库时，使用项目的领域术语表，确保测试名称和接口词汇与项目语言一致，并尊重你正在触及的领域的 ADR。

写任何代码之前：

- [ ] 与用户确认需要什么接口变更
- [ ] 与用户确认要测试哪些行为（排优先级）
- [ ] 识别 [deep modules](deep-modules.md) 的机会（小接口，深实现）
- [ ] 为[可测试性](interface-design.md)设计接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对方案的批准

问：「公共接口应该是什么样的？哪些行为最需要测试？」

**你不可能测试所有东西。** 与用户确认哪些行为最重要。把测试精力集中在关键路径和复杂逻辑上，而不是每个可能的边界情况。

### 2. 追踪弹

写一个测试来确认系统的一件事：

```
RED：   为第一个行为写测试 → 测试失败
GREEN：写最少的代码使其通过 → 测试通过
```

这是你的追踪弹——证明这条路径端到端是通的。

### 3. 增量循环

对于每个剩余的行为：

```
RED：   写下一个测试 → 失败
GREEN：最少代码使其通过 → 通过
```

规则：

- 每次一个测试
- 只写刚好让当前测试通过的代码
- 不要预见未来的测试
- 保持测试聚焦于可观察的行为

### 4. 重构

所有测试通过后，寻找[重构候选](refactoring.md)：

- [ ] 提取重复
- [ ] 加深模块（将复杂性移到简单接口后面）
- [ ] 在自然的地方应用 SOLID 原则
- [ ] 思考新代码揭示了现有代码的什么问题
- [ ] 每次重构后运行测试

**绝不在 RED 状态下重构。** 先到 GREEN。

## 每轮检查清单

```
[ ] 测试描述的是行为，不是实现
[ ] 测试只使用公共接口
[ ] 测试能经受住内部重构
[ ] 代码是针对这个测试的最少实现
[ ] 没有添加推测性功能
```

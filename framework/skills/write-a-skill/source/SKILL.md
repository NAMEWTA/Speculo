# 编写技能

## 流程

1. **收集需求** —— 向用户了解：
   - 技能覆盖什么任务/领域？
   - 应该处理哪些具体用例？
   - 需要可执行脚本还是只需要指令？
   - 有没有参考资料要包含？

2. **起草技能** —— 创建：
   - SKILL.md，包含简洁的指令
   - 如果内容超过 500 行，添加额外的参考文件
   - 如果需要确定性操作，添加工具脚本

3. **与用户审查** —— 展示草稿并询问：
   - 这覆盖了你的用例吗？
   - 有什么遗漏或不清楚的吗？
   - 哪些章节应该更详细/更简洁？

## 技能结构

```
skill-name/
├── SKILL.md           # 主要指令（必需）
├── REFERENCE.md       # 详细文档（按需）
├── EXAMPLES.md        # 使用示例（按需）
└── scripts/           # 工具脚本（按需）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力的简要描述。当 [具体触发条件] 时使用。
---

# 技能名称

## 快速开始

[最小可工作示例]

## 工作流

[复杂任务的分步流程和检查清单]

## 高级功能

[链接到单独的文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是**你的 Agent 在决定加载哪个技能时看到的唯一内容**。它与其他已安装的技能一起展示在系统提示中。你的 Agent 阅读这些描述，然后根据用户的请求选择相关的技能。

**目标**：给你的 Agent 刚好足够的信息，让它知道：

1. 这个技能提供什么能力
2. 何时/为什么触发它（具体关键词、上下文、文件类型）

**格式**：

- 最多 1024 个字符
- 用第三人称写
- 第一句：做什么
- 第二句：「当 [具体触发条件] 时使用」

**好的例子**：

```
从 PDF 文件中提取文本和表格，填写表单，合并文档。当处理 PDF 文件或用户提到 PDF、表单或文档提取时使用。
```

**坏例子**：

```
帮助处理文档。
```

坏例子让你的 Agent 无法将这个技能与其他文档类技能区分开来。

## 何时添加脚本

在以下情况下添加工具脚本：

- 操作是确定性的（验证、格式化）
- 同样的代码会被反复生成
- 错误需要显式处理

脚本比生成的代码更省 token，也更可靠。

## 何时拆分文件

在以下情况下拆分为单独的文件：

- SKILL.md 超过 100 行
- 内容包含不同的领域（财务 vs 销售 Schema）
- 高级功能很少用到

## 审查清单

起草后，检查：

- [ ] 描述包含触发条件（「当……时使用」）
- [ ] SKILL.md 在 100 行以内
- [ ] 没有时效性信息
- [ ] 术语一致
- [ ] 包含具体示例
- [ ] 引用只有一层深度

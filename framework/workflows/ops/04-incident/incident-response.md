# Incident Response — 事故响应规范

本阶段是事故的"黄金一小时"操作手册。优先级：**止损 > 隔离 > 根因 > 学习**。AI 在执行 `phase=triage` 时读本文件作为应急流程。

## 一、上下文与产物

- **输入**：触发源（`../03-monitor/` 的回滚信号 / 外部告警 / 用户反馈）；相关 change 的 deploy 与 rollback 产物；项目 LESSONS.md
- **产物**：`incident.md`（基于 `../_templates/incident-template.md`）
- **隔离原则**：事故产物写入独立 change 目录 `<YYYY-MM-DD>-incident-<topic>/`；不与触发它的 dev/ops change 混在一起

## 二、章节填写引导

### 事件摘要
- **发生时间**：检测到时间（不是发生时间，发生时间留给根因段）
- **检测来源**：监控告警 / 用户反馈 / 主动巡检 任一
- **影响面**：受影响的用户范围、服务范围、数据范围（用数字量化）
- **当前状态**：调查中 / 止损中 / 已止损 / 已恢复

### 严重度分级
| 等级 | 标准 | 升级条件 |
|------|------|---------|
| `P0` | 全站不可用 / 核心业务中断 / 数据安全风险 / 监管合规风险 | 任一即 P0 |
| `P1` | 关键功能受损但有降级 / 部分用户群体不可用 / 性能严重下降 | 影响 ≥20% 用户或营收路径 |
| `P2` | 非关键功能问题 / 局部体验下降 | 仅 P2 时不强制 postmortem |

分级一旦设定，原则上**只升不降**；后续证据要求降级，需在 incident.md 文末"## 降级决策"段说明理由 + 时间戳。

### 止损动作
按时间顺序记录，每条含：

| 时间 | 动作 | 执行人 | 命令输出摘要 | 结果 |
|------|------|--------|------------|------|

常见动作（按推荐顺序）：
1. **回滚**：执行 `../02-deploy/deploy-rollback.md`（如触发自部署）
2. **切流**：把流量切到健康节点 / 旧版本
3. **降级**：关闭非关键功能 / 切到本地兜底
4. **限流**：减缓上游流量

严重度 P0：必须有至少一条已执行的回滚 / 切流 / 降级动作（不允许"还在分析"）。

### 根因分析（5 Whys）
- **即时原因**（Immediate cause）：直接引发的代码 / 配置 / 操作变更
- **中间原因**（Intermediate cause）：为什么这个变更通过了之前的拦截（评审？测试？灰度？）
- **根本原因**（Root cause）：组织 / 流程 / 工具层面的系统性问题

每层用证据支撑（代码 diff / 日志 / commit hash / 监控截图引用）。

## 三、与回滚预案的衔接

若触发源是 `../03-monitor/` 的 `monitor_result=rollback-triggered`：
- 优先执行 `../02-deploy/deploy-rollback.md` 步骤
- 把回滚步骤的实际输出贴入 `incident.md "止损动作"段
- 回滚成功后继续根因分析（回滚是止损，不是终点）

若事故源不是部署：
- 跳过回滚段
- 在止损动作选择切流 / 降级 / 限流

## 四、LESSONS 写回（强制收尾）

事故结束前必须把抽象失败模式追加到 `../../../.speculo/.config/LESSONS.md`：

> "X 类变更（如 schema 迁移）在 Y 条件（如未做 down 预演）下导致 Z 后果（如回滚失败）。下次 dev/05-review 必须检查 down 预演输出。"

要求：
- 用"X 在 Y 导致 Z"模板写
- 显式给"下次哪个 phase 该 grep / 检查什么"的指引
- 不写本次事故的细节（细节归 incident.md），只写可复用的模式

## 五、完成准则（机器可验证）

- `grep -c '\[TODO:' incident.md` = 0
- 文件含 `## 事件摘要` / `## 严重度分级` / `## 止损动作` / `## 根因分析` 四个标题
- 严重度字段为 `P0` / `P1` / `P2` 三者之一
- "止损动作" 段含 ≥1 行表格记录
- 严重度 P0 时：止损动作含"回滚" / "切流" / "降级" 关键词至少一个，且贴有命令输出
- "根因分析" 段含 `即时原因` / `中间原因` / `根本原因` 或等价三层结构
- `incident_severity` / `triage_actions` / `root_cause` / `lessons_appended` 写入 `.status.json`
- `lessons_appended=true` 才允许 phase 标 `completed`

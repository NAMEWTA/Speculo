# Deploy Rollback — 回滚预案规范

本阶段产出一份**量化触发、机械可执行、有回滚后验证**的预案文件。每次部署都必须产出本文件，不允许"等出事再想"。AI 在执行 `phase=rollback-plan` 时读本文件作为填写指引。

## 一、上下文与产物

- **输入**：本 change 的 `deploy.md`、`design-arch.md`（理解模块边界）、`implement.md`（是否含 schema 变更）
- **产物**：`deploy-rollback.md`（基于 `../_templates/deploy-rollback-template.md`）
- **复用方**：`../03-monitor/` 与 `../04-incident/` 工作流硬依赖本预案

## 二、章节填写引导

### 触发条件
必须**量化**，每条形如：

> 指标 `<指标名>` 在窗口 `<时间>` 内 `<阈值条件>`

示例：
- 错误率 >2% 持续 5 分钟
- P99 延迟 >800ms 在 5 次连续 1 分钟采样中
- 健康检查失败率 >50%
- 业务关键指标（订单创建成功率）跌幅 >10%

不接受的触发条件：
- "感觉不对劲"
- "用户投诉变多"（除非有量化定义）
- "运维觉得需要回滚"

### 回滚步骤
按顺序列出可执行命令，每条含：
- 命令本身（可粘贴）
- 预期输出
- 失败时的下一步

涉及 schema 变更时，必须包含：
- down 迁移命令（如 `pnpm migrate:down` 或 `alembic downgrade -1`）
- 数据兼容性说明（旧版本是否能读新 schema 写入的数据）
- 数据丢失风险评估（哪些字段会丢、哪些会被默认值替代）

### 回滚后验证
至少 2 项验证：

| 验证项 | 方法 | 通过条件 |
|--------|------|---------|
| 健康检查 | `curl <health-endpoint>` | 200 OK 持续 3 次 |
| 关键业务接口 | 复现 `prd.md` 中典型 AC 流程 | 行为符合预期 |
| 监控指标恢复 | 对照触发条件指标 | 恢复到部署前 baseline |

## 三、与监控/事故工作流的衔接

- `../03-monitor/`：观测期间命中触发条件 → 自动或人工触发本预案
- `../04-incident/`：事故响应时，根据严重度直接复用本预案，不重新设计

## 四、Schema 回滚强约束

若 `deploy.md` 中 `schema_migrated=true`：
- 必须有 down 迁移命令（不允许"目前没法回滚 schema"）
- 必须有"已在本地或 staging 预演 down 迁移"的输出贴入
- 必须评估"回滚后数据兼容性"（旧版本能否消费迁移后未清理的数据）
- 不可逆 schema 变更（如删字段）：必须在 `implement.md` 实施时就规划"应用层兼容旧字段缺失"，使回滚不依赖 schema down

## 五、回滚演练（推荐）

非生产环境建议每次部署后跑一次完整回滚演练；生产环境每季度至少一次。演练结果追加到 `../../../.speculo/.config/LESSONS.md`。

## 六、完成准则（机器可验证）

- `grep -c '\[TODO:' deploy-rollback.md` = 0
- 文件含 `## 触发条件` / `## 回滚步骤` / `## 回滚后验证` 三个标题
- "触发条件" 段含 ≥1 个量化阈值（数字 + 单位 + 时间窗口）
- "回滚步骤" 段含 ≥1 个 ` ``` ` 代码块
- 若 `schema_migrated=true`：必须含 "down 迁移" 或等价关键词
- "回滚后验证" 段含 ≥2 项验证（表格 ≥2 行或等价列表）
- `rollback_ready=true` 与 `rollback_triggers` 数组写入 `.status.json`

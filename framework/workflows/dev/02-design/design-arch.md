# Design Arch — 架构设计规范

本阶段输出**技术架构决策与跨模块交互蓝图**。AI 在执行 `phase=arch` 时读本文件作为填写指引。

## 一、上下文与产物

- **输入**：同 change 下的 `prd.md`；项目根 `../../../.speculo/.config/{ARCHITECTURE.md, RULES.md, DATA-MODEL.md, LESSONS.md}`（如存在）
- **产物**：`design-arch.md`（基于 `../_templates/design-arch-template.md`）
- **角色红线**：本阶段**只产架构与决策记录，不直接写实现代码**（实现归 `../03-implement/`）

## 二、章节填写引导

### 整体架构
- 列出本次需求涉及的**全部模块/组件**：内部新增、内部修改、外部依赖三类分开
- 至少 1 张架构图：mermaid `graph TD` 或 ascii 框图，标注模块名 + 关键调用箭头
- 用 1 段文字补全图无法表达的语义（例如：哪些是无状态、哪些是关键路径）

### 关键决策（ADR-style）
每条决策包含 4 要素：

> **ADR-N · 标题**
>
> - **背景**：触发本决策的需求/约束（引用 `prd.md` 章节）
> - **选定方案**：选了什么
> - **备选方案**：至少 1 个被拒方案 + 拒绝理由（拒绝理由禁止"性能更好"这类空话，要给场景）
> - **影响**：对其他模块、迁移、回滚的连带影响

至少 2 条决策。复杂 change 建议 ≥4 条。

### 数据流
- 跨模块的输入/输出方向（用 mermaid `sequenceDiagram` 或 1-2 段文字描述）
- 标注同步 vs 异步、是否含敏感数据（PII、token、密钥）
- 列出涉及的**数据库表/集合改动**：新增 / 修改 schema / 数据迁移，每项标"是否需要可逆迁移"

## 三、宪法 Gate 自检（强约束）

在产物完成前对照 `../../../.speculo/.config/RULES.md`（若存在）逐条检查：

- 触发 RULES.md 中"禁止使用的 API/库"？
- 违反命名约定、目录结构、依赖白名单？
- 触发数据安全、依赖管理或破坏性变更高门槛条款？

任何违规：
- **首选**：调整方案绕开
- **不可绕开时**：在 `design-arch.md` 文末新增"## 宪法例外声明"段，写"违反条款 X / 原因 Y / 用户审批 Z"

## 四、反幻觉与反重复

- 引用任何外部库 / 协议 / API 字段名前，必须在已知文档或代码库 grep 验证存在性，命中后才写入决策
- 引用项目内部抽象前，先 grep 同类实现：若已存在，决策应"复用 + 必要时扩展"而非"新造"
- 不允许"应该可以工作"式断言；任何不确定项标 `[NEEDS-VERIFY: ...]`

## 五、与 PRD 的映射

文末必须有"## PRD 映射"段（不存在则新增），表格形式：

| PRD 章节/AC | 本设计对应位置 | 备注 |

确保每条核心需求都能在架构里找到承载位置。

## 六、完成准则（机器可验证）

- `grep -c '\[TODO:' design-arch.md` = 0
- `grep -c '\[NEEDS-VERIFY:' design-arch.md` = 0
- 文件含 `## 整体架构` / `## 关键决策` / `## 数据流` 三个标题
- "关键决策"段内 `ADR-` 计数 ≥2，每条含"备选方案"或"被拒"
- "整体架构"段含 ≥1 个 ` ``` ` 代码块（mermaid 或 ascii）
- 文末含 "PRD 映射" 段，且表格非空
- 若 RULES.md 存在且本设计有违规：必须含 "宪法例外声明" 段
- `arch_decisions` 字段写入 `.status.json`，条目数与 ADR 条目数一致

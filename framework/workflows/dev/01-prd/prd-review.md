# PRD Review — 评审阶段规范

本阶段对 Core 阶段产出的 `prd.md` 做一次**结构化合规审查**，输出可执行的修订指令或放行结论。AI 在执行 `phase=review` 时读本文件作为评审手册。

## 一、上下文与产物

- **输入**：同 change 下的 `prd.md`
- **产物**：`prd-review.md`（基于 `../_templates/prd-review-template.md`）
- **角色红线**：评审通道**只产报告与修订建议，不直接改 `prd.md`**；修订动作回 Core 阶段完成。

## 二、评审检查矩阵（逐项执行）

| 维度 | 检查项 | 通过条件 |
|------|--------|----------|
| 完整性 | 四个章节齐全 | `prd.md` 含 `项目背景` / `核心需求` / `范围与非范围` / `验收标准` 四个 `## ` 标题 |
| 完整性 | 无残留占位符 | `grep -c '\[TODO:' prd.md` = 0 且 `grep -c '\[NEEDS-CLARIFY:' prd.md` = 0 |
| 可测性 | 每条 AC 可被测试覆盖 | 每条验收标准能转化为单测、e2e 或人工 UAT 至少一种 |
| 可测性 | 用户故事数量与质量 | ≥3 条，含角色/能力/价值三段 |
| 范围 | 非范围条目非空 | "范围与非范围"段至少 1 条"非范围" |
| 范围 | 无 HOW 渗透 | 不出现具体技术栈名、库名、API 端点（这些归 `../02-design/`） |
| 一致性 | AC 反向引用用户故事 | 每条 AC 显式映射到 ≥1 个 `US-N` |
| 一致性 | 与 RULES.md 不冲突 | 对照 `../../../.speculo/.config/RULES.md`（如存在），需求未触发禁用项 |
| 风险 | 干系人与优先级齐全 | `.status.json` 中 `stakeholders` 非空、`priority` 已选 |

## 三、评审决议三档

- **通过**：所有检查项通过；可直接进入下游 `../02-design/`
- **通过-小修**：有 ≤3 条非阻塞修订意见；修订动作明确，无需重审
- **不通过**：≥1 条阻塞项（如缺失整段、AC 不可测、范围未切分）；必须回 Core 阶段重写后重审

## 四、评审意见格式

每条意见三要素：

> **位置** · 严重度 · 修订指令
>
> 例：`prd.md#核心需求 US-2` · `block` · "缺少价值描述：补全'以便 ...'"

严重度三档：`block`（阻塞，必修）/ `major`（建议必修，否则降级为"通过-小修"）/ `minor`（提示性）。

## 五、二次评审

当决议为"通过-小修"且 Core 阶段已完成修订，可在同 `prd-review.md` 文件追加"## 二次评审记录"段（不另起文件），列出原意见对应处理结果（已修订 / 部分修订 / 拒绝并理由）。

## 六、写回 LESSONS（可选）

若评审发现"本次问题与某历史失败模式相同"，把简短归纳追加到 `../../../.speculo/.config/LESSONS.md`，避免下次重蹈（与 `../05-review/` 的 LESSONS 写回闭环统一）。

## 七、完成准则（机器可验证）

- `prd-review.md` 顶部"评审决议"字段为 `通过` / `通过-小修` / `不通过` 三者之一
- 若决议非"通过"，"评审意见"段内 `block` 或 `major` 计数 ≥1
- 若决议为"通过-小修"，模板中"待修订项"段非空且每条标注严重度
- 把 `prd_review_decision` 写入对应 change 的 `.status.json`
- 决议为"通过"或"通过-小修+已二次评审"才允许把 `phase_history` 中 review 条目置 `completed`

# PRD Core — 核心内容填写规范

本阶段把模糊诉求转化为**面向交付**的结构化需求文档。AI 在执行 `phase=core` 时读本文件作为填写指引。

## 一、上下文与产物

- **输入**：用户的原始诉求（任何形式：一句话、bug 报告、产品想法、竞品对比、用户访谈纪要）
- **产物**：`prd.md`（基于 `../_templates/prd-template.md`）
- **写入位置**：`../../../.speculo/dev/<change-name>/prd.md`

## 二、章节填写引导

### 项目背景
回答三件事：**问题是什么 / 谁感到痛 / 不做的代价**。
- 用真实场景或数据描述痛点（避免"会更好"这种空话）
- 指明目标用户群体（角色、规模、使用频率）
- 一句话给出"做完后预期改善的指标方向"（不需要精确数值，但要可观察）

### 核心需求
列 **3-5 条用户故事**，格式：

> 作为 `<角色>`，我希望 `<能力>`，以便 `<价值>`。

要求：
- 每个故事独立可测（删任意一条，其他仍构成可交付增量）
- 标注优先级 `P1` / `P2` / `P3`（P1 = MVP 必含）
- 避免技术名词渗透到故事（不出现"用 Redis 缓存"这类实现细节）

### 范围与非范围
- **范围**：本次 PRD 明确包含的能力清单（与核心需求一一对应或更细化）
- **非范围**：本次明确**不做**的事（至少 1 条；目的是切断范围蔓延的口子）
- 非范围条目格式："本次不做 X，因为 Y"（理由是后续争议时的依据）

### 验收标准
每条满足 **Given / When / Then** 三要素：

> **AC-1**：**Given** 用户已登录且购物车空 / **When** 添加任一商品 / **Then** 购物车计数 ≥1 且本地缓存更新

要求：
- 每条 AC 至少对应一条核心需求（用 `US-N` 反向引用）
- 描述用户可感知的可观察结果（不写"调用 API X"）
- 验证方式可选：自动化测试命令 / 手动 UAT 编号 / 二者皆可

## 三、歧义处理规则

任何**填写过程中遇到的歧义**：
- 不允许 AI 凭直觉猜测
- 在原位置插入 `[NEEDS-CLARIFY: <具体问题>]` 标记
- 在文末"待澄清项"段（如不存在则新增）汇总编号
- 标记必须在 Review 前解决或被用户显式接受

> 与模板里的 `[TODO: ...]` 占位符语义区分：`[TODO:]` 表示"用户/AI 知道要填什么、按指引填即可"；`[NEEDS-CLARIFY:]` 表示"信息不足，必须先问/查"。

## 四、参考与禁忌

- **可参考**：`../../../.speculo/.config/STRUCTURE.md`、`DATA-MODEL.md`、`LESSONS.md` 中与本需求同主题的历史教训
- **禁忌**：
  - 写 HOW（怎么实现）— 留给 `../02-design/`
  - 引入未经验证的外部依赖名（先标 `[NEEDS-CLARIFY:]`）
  - 复制竞品文档原文（要内化为本项目语境）

## 五、完成准则（机器可验证）

- `grep -c '\[TODO:' prd.md` 输出 0
- `grep -c '\[NEEDS-CLARIFY:' prd.md` 输出 0
- 文件包含且仅包含以下四级 `## ` 标题：`项目背景` / `核心需求` / `范围与非范围` / `验收标准`
- `核心需求` 段内"作为"出现次数 ≥3
- `范围与非范围` 段内"非范围"或"本次不做"出现次数 ≥1
- `验收标准` 段内"Given"出现次数 ≥1（或等价"前置 / 触发 / 结果"三段结构）
- 把对应字段写入 `.status.json`：`stakeholders` 非空、`priority` ∈ {P0,P1,P2,P3}、`clarifications_open=0`

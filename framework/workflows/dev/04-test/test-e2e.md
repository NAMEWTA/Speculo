# Test E2E — 端到端测试规范

本阶段为单测无法覆盖的**跨组件 / 跨服务 / 跨用户操作场景**建立端到端测试，并对性能、安全、兼容、可观测四维度作显式状态声明。AI 在执行 `phase=e2e` 时读本文件作为填写指引。

## 一、上下文与产物

- **输入**：同 change 下的 `prd.md`、`design-api.md`、`test-unit.md`（已覆盖 AC 清单）
- **产物**：`test-e2e.md`（基于 `../_templates/test-e2e-template.md`）
- **覆盖目标**：补全单测尚未覆盖的 PRD AC；并对四维度（性能/安全/兼容/可观测）作"覆盖 / 不适用 / 跳过+理由"声明

## 二、章节填写引导

### 测试场景
每个场景含三要素：

> **场景 SCN-N · 标题**（关联 AC-X）
>
> - **前置条件**：环境状态 / 数据准备 / 用户身份
> - **执行步骤**：编号步骤，每步可机械执行（无歧义）
> - **验收点**：可观察结果（页面元素 / 接口返回 / 日志条目 / 指标变化）

要求：
- 每个场景关联至少一条 PRD AC（反向引用 `AC-N`）
- 自动化已覆盖的场景，附执行命令；手动 UAT 用 UAT-N 编号

### 环境配置
- 测试运行所需的环境（域名、端口、依赖服务版本）
- 外部依赖 mock 策略（若引入 mock，说明 mock 边界 + 真实环境差异）
- 测试数据准备方式（fixture / seed / 自动生成）

### 已知限制
- 暂未自动化覆盖的场景及人工 UAT 编号
- 故意不覆盖的场景及理由（如"上游服务限流测试需 SRE 协助"）
- 若全部覆盖：显式写"无未覆盖场景"

## 三、五维度状态声明（强制段）

新增"## 五维度覆盖状态"段（不在模板默认结构内但本规范强制）：

| 维度 | 状态 | 证据 / 跳过理由 |
|------|------|----------------|
| 功能（functional） | covered / skipped / n/a | 关联 AC 与场景 |
| 性能（performance） | covered / skipped / n/a | 工具与基线（如 Lighthouse / load test） |
| 安全（security） | covered / skipped / n/a | 工具与发现（SAST / 依赖漏扫 / OWASP 类别） |
| 兼容（compat） | covered / skipped / n/a | 浏览器 / OS / 数据迁移版本矩阵 |
| 可观测（observability） | covered / skipped / n/a | 日志 / 指标 / 告警是否就绪 |

- `covered`：有具体测试或人工 UAT 证据
- `skipped`：当前 change 选择跳过，**必须**给理由（不接受空）
- `n/a`：本 change 不涉及该维度（如纯文档改动跳过性能维度）

## 四、UAT 脚本（手动验收）

每个 UAT 含：
- **UAT-N · 场景标题**（关联 AC-X / SCN-Y）
- **前置**：环境状态
- **步骤**：编号步骤
- **期望**：可观察结果
- **实际**：通过 / 失败 / 描述
- **执行人 / 时间**：（具体姓名 + ISO 时间戳）

至少 1 条 UAT 脚本（即使全部自动化，也保留 1 条作为"人感"兜底）。

## 五、与 test-unit.md 的衔接

- 单测已覆盖的 AC 不需重复编 e2e 场景，但 e2e 必须显式声明"AC-X 已由 test-unit.md 覆盖"避免漏检
- 单测无法覆盖的 AC（涉及跨服务/UI 流程/真实环境）→ e2e 必须建场景

## 六、完成准则（机器可验证）

- `grep -c '\[TODO:' test-e2e.md` = 0
- 文件含 `## 测试场景` / `## 环境配置` / `## 已知限制` 三个标题
- 含 `## 五维度覆盖状态` 段，表格 ≥5 行
- 含 `UAT-` 编号至少 1 个
- 每个未由 test-unit.md 覆盖的 PRD AC 在测试场景段出现
- `e2e_pyramid` 五字段写入 `.status.json`，每字段值 ∈ {covered, skipped, n/a}
- `ac_coverage` 中所有 AC 状态非 `uncovered`，或显式声明"已知未覆盖 + 接受理由"
